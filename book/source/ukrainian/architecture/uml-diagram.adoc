[#uml-diagram]
=== UML Diagram

image::architecture/uml-diagram.jpg[UML Diagram, align="center"]

[[uml-definition]]*UML(Unified Modeling Language)* — це стандартизована мова моделювання, яка використовується для візуалізації, специфікації, конструювання та документування програмних систем. Вона надає набір графічних елементів і діаграм, які допомагають розробникам і архітекторам систем краще розуміти структуру та поведінку програмного забезпечення.

[[uml-diagram-types]]
Розрізняють два основні типи діаграм UML:

* *Стркуктурні діаграми* — описують статичну структуру системи, включаючи класи, об'єкти, компоненти та їх взаємозв'язки. Приклади: діаграми класів, діаграми об'єктів, діаграми компонентів;
* *Поведенкові діаграми* — описують динамічну поведінку системи, включаючи потоки управління, взаємодії між об'єктами та стани. Приклади: діаграми послідовностей, діаграми діяльності, діаграми станів.

[[uml-diagram-behavioral-diagram-types]]
До Поведенкових діаграм відносяться:

* <<uml-activity-diagram,*Діаграма діяльності (Activity Diagram)*>> — відображає потоки робіт або процесів у системі, що допомагає у моделюванні бізнес-процесів;
* <<uml-communication-diagram,*Діаграма комунікацій (Communication Diagram)*>> — відображає взаємодії між об'єктами з акцентом на їхні зв'язки;
* <<uml-interaction-overview-diagram,*Диаграмма обзора взаимодействия (Interaction overview diagram)*>> — комбінує елементи діаграм діяльності та послідовностей для надання огляду складних взаємодій;
* <<uml-sequence-diagram,*Діаграма послідовностей (Sequence Diagram)*>> — ілюструє взаємодії між об'єктами у вигляді послідовності повідомлень, що допомагає у розумінні логіки роботи системи;
* <<uml-state-machine-diagram,*Діаграма станів (State Machine Diagram)*>> — показує стани об'єктів і переходи між ними, що допомагає у моделюванні поведінки об'єктів;
* <<uml-timing-diagram,*Діаграма часових діаграм (Timing Diagram)*>> — ілюструє зміни станів об'єктів у часі, що допомагає у моделюванні реального часу;
* <<uml-use-case-diagram,*Діаграма випадків використання (Use Case Diagram)*>> — показує функціональні можливості системи з точки зору користувачів і їх взаємодії з системою.


[[uml-diagram-structural-diagram-types]]
До Структурних діаграм відносяться:

* <<uml-class-diagram,*Діаграма класів (Class Diagram)*>> — показує класи системи, їх атрибути, методи та взаємозв'язки між ними. Це основний інструмент для моделювання об'єктноорієнтованих систем;
* <<uml-component-diagram,*Діаграма компонентів (Component Diagram)*>> — ілюструє організацію та залежності між різними компонентами системи, що допомагає у плануванні архітектури програмного забезпечення;
* <<uml-composite-structure-diagram,*Диаграмма композитной структуры (Composite structure diagram)*>> — відображає внутрішню структуру класів і компонентів, включаючи їхні частини та зв'язки;
* <<uml-deployment-diagram,*Діаграма розгортання (Deployment Diagram)*>> — показує фізичне розгортання артефактів на вузлах, що допомагає у плануванні інфраструктури;
* <<uml-object-diagram,*Діаграма об'єктів (Object Diagram)*>> — відображає конкретні екземпляри класів і їх зв'язки в певний момент часу;
* <<uml-composite-package-diagram,*Діаграма пакетів (Package Diagram)*>> — організовує елементи моделі в пакети, що допомагає у структуризації великої системи;
* <<uml-profile-diagram,*Діаграма профілів (Profile Diagram)*>> — дозволяє створювати спеціалізовані розширення UML для конкретних доменів або платформ.

[[uml-diagram-usage-sequence]]
Послідовність використання діаграм UML:

1. Визначення вимог: Використання Use Case діаграм для збору та аналізу вимог від зацікавлених сторін;
2. Моделювання структури: Використання Class діаграм для визначення основних класів, їх атрибутів та методів;
3. Моделювання поведінки: Використання Sequence та Activity діаграм для опису взаємодій між об'єктами та потоків робіт;
4. Деталізація: Використання State Machine діаграм для моделювання поведінки окремих об'єктів у різних станах;
5. Організація: Використання Package діаграм для структуризації моделі у вигляді пакетів;
6. Планування розгортання: Використання Deployment діаграм для визначення фізичного розгортання системи;
7. Документування: Використання Profile діаграм для створення спеціалізованих розширень UML, якщо необхідно.
