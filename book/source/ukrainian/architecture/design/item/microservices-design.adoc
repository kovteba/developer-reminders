// [[microservices-architecture-definition]] Microservices Architecture - це архітектурний стиль, в якому програмне забезпечення складається з невеликих незалежних сервісів, які співпрацюють один з одним, використовуючи легковагові механізми, такі як HTTP або REST API. Кожен сервіс може бути написаний на різних мовах програмування та може використовувати різні технології для зберігання даних.
//
// [[microservices-architecture-communication]]
// В мікросервісній архітектурі використовується API-інтерфейси RESTful, служба повідомлень (JMS) або потокова передачи «видавець-передплатник» (pub/subs), що не потребує  щоб мікросервіси підтримували активне з'єднання при обміні даних.
//
// API - є інструментом для архітектури мікросервісів. Що дозволяє обмінюватися повідомленями напряму між мікросервісамі без застосування якогось централізованого каналу. Однак це може приводити до складних шляхів спілкування між великою кількосттю мікросервісів
//
// [[microservices-architecture-advantages]] Переваги мікросервісної архітектури:
//
// * <<microservices-architecture-advantages-scaling,Масштабованість>>
// * <<microservices-architecture-advantages-technology-diversity,Різноманітність технологій>>
// * <<microservices-architecture-advantages-fault-isolation,Ізоляція помилок>>
// * <<microservices-architecture-advantages-ease-of-deployment,Легкість розгортання>>
// * <<microservices-architecture-advantages-continuous-delivery,Постійна доставка>>
//
// [[microservices-architecture-advantages-scaling]]
// Мікросервіси можуть бути масштабовані незалежно один від одного. Це означає, що ви можете масштабувати лише ті мікросервіси, які потребують більше ресурсів, а не всю систему.
//
// [[microservices-architecture-advantages-technology-diversity]]
// Ви можете використовувати різні технології для різних мікросервісів. Це означає, що ви можете використовувати найкращі технології для кожного мікросервісу.
//
// [[microservices-architecture-advantages-fault-isolation]]
// Якщо один мікросервіс виходить з ладу, це не впливає на інші мікросервіси. Це означає, що ваша система залишається працездатною, навіть якщо один з мікросервісів виходить з ладу. Також така архітектура дозволяє більш єксперементувати з новими технологіями знаючи що вихід з ладу одного сервісу не вплине на роботу інших.
//
// [[microservices-architecture-advantages-ease-of-deployment]]
// Ви можете розгортати кожен мікросервіс незалежно від інших мікросервісів. Це означає, що ви можете розгортати нові функції швидше і без ризику впливу на інші мікросервіси.
//
// [[microservices-architecture-advantages-continuous-delivery]]
// Ви можете постійно впроваджувати нові функції, оновлення та виправлення помилок без великих зусиль. Це означає, що ваша система завжди залишається актуальною та безпечною. Мікросервісна архітектура ідеально підходить для використання в CI/CD.
//
// Також можна окремо відмітити що мікросервісна архітектура має більш велику доступність до розуміння, тому що новим розробникам не треба вивчати всю кодову базу а тільки мікросервіс з яким працюють. Також одною з головних особливостей є те що код можна перевикористовувати. Також для мікросервісногї архітектури властива довговічність так як по факту нема єдиної точки відмови.
//
// [[microservices-architecture-disadvantages]]
// Недоліки мікросервісної архітектури:
//
// * <<microservices-architecture-disadvantages-distributed-system,Складна розподілена система>>
// * <<microservices-architecture-disadvantages-network-latency,Затримка мережі>>
// * <<microservices-architecture-disadvantages-data-consistency,Неспівмірність даних>>
// * <<microservices-architecture-disadvantages-monitoring,Моніторінг>>
//
// [[microservices-architecture-disadvantages-distributed-system]]
// Мікросервіси можуть бути розподілені на різних серверах, що може ускладнити відлагодження та моніторинг системи.
//
// [[microservices-architecture-disadvantages-network-latency]]
// Коли мікросервіси співпрацюють один з одним через мережу, це може призвести до затримок у відповідях.
//
// [[microservices-architecture-disadvantages-data-consistency]]
// Коли дані розподілені між різними мікросервісами, це може призвести до проблем з консистентністю даних. Для мінімізації цього єфекту треба мінімізувати кількість сервісів які взаємодіють з базою даних.
//
// [[microservices-architecture-disadvantages-monitoring]] Зважаючи на те що у реальній системі кількість мікросервісів може бути дуже велика, та ще й самі мікросервіси можуть бути розподілені на різних серверах, моніторинг системи може бути складним завданням.
//
// [[microservices-architecture-soa-problems]]Мікросервісна архітектура дозволяє усунути такі проблеми з <<service-oriented-architecture,SOA>> архітектури як <<service-oriented-architecture-esb,ESB>>
//
// Мікросервісна архітектура це  найкращій варіант для гнучкої розробки, що дозволяє швидко додавати зміни в код і хвилюючись за стабільнічсть додатку використовую інструменти CI/CD. Ця архітектура є найкращим вибіром  якщо треба використовувати різні мови розробки6 фреймворки або бібліотеки, використовувати різні бази даних, я також при необхідності маштабування окремих сервісів.
//
// [[microservices-architecture-vs-monolit]]
//
// С одной стороны, когда монолитные архитектуры служат в качестве крупномасштабной системы, это может усложнить ситуацию. Они могут быть громоздкими в работе, когда нужно добавить новые функции, внести изменения или даже удалить некоторые ненужные функции.
//
// image::image-2024-12-12-15-21-33-621.png[]
//
// Однако монолитные архитектуры по-прежнему являются хорошим вариантом для приведенных ниже случаев:
//
// * малогабаритные приложения;
// * нет необходимости в обновлении технологии;
// * вам требуется меньше времени для вывода на рынок;
// * команды знакомы с Монолитными подходами.
//
// С другой стороны, архитектура микросервисов не очень хорошо подходит для небольших приложений, требующих единого технологического стека, платформы деплоя, доменной зоны и т. д. Но для крупномасштабных приложений пока нет ничего лучше этой архитектуры.
//
// image::image-2024-12-12-15-21-56-117.png[]
//
// Микросервисная архитектура является хорошим вариантом для следующих случаев:
//
// * широкомасштабные приложения;
// * вам необходимо обновление технологии;
// * разные команды работают с разными техническими скилами (языки программирования, фреймворки и другое);
// * если у вас есть различные домены, которые необходимо обработать;
// * И многие другие.
