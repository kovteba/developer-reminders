[#service-oriented-architecture]
==== Service-Oriented architecture
// TODO Service-Oriented architecture

// [[service-oriented-architecture-definition]]Service-Oriented architecture - це підхід до розробки програмного забезпечення, який використовує сервіси як основний елемент для побудови різних програмних систем. Цей підхід дозволяє створювати програмні системи з набору сервісів, які можуть бути використані в різних програмних системах. Кожен сервіс повинен бути повністю незалежним від інших сервісів, тобто він повинен мати свою власну логіку, дані та інші ресурси. SOA охоплює більш широку діапазон дій на відміну від мікросервісів. У той час як мікросервіси мають більш вузьку спеціалізацію. Наприклад керування складом в цілому це буде SOA підхід. Для того зоб це перетворити у мікросервіси можна розділити цю задачу на кілька менших задач, наприклад перевірка наявності товару, замовлення товару, відправка товару. Тоб то у мікросервісному підході сервіс має тільки один сенс існування тобто тільки одну задачу
// 
// У SOA є таке поняття як ESB (Лінійка корпоративних сервісів). На ESB можна дивитись як на централізований сервіс, оркестратор який займається маршрутизацією вхідні запроси на відповідний сервіс і преобразує запроси у формат який приймає відподідний сервіс. Але це створює додаткову точку входу в систему фбо точкою "відмови", яка може стати проблемою при великій кількості сервісів.
// 
// Не існує чітко визначених стандартів реалізації SOA, є декілька рекомендацій:
// 
// * <<service-oriented-architecture-weak-coupling, Слабке зчеплення>>
// * <<service-oriented-architecture-abstraction, Абстрагування>>
// * <<service-oriented-architecture-level-of-detail, Ступінь деталізації>>
// 
// [#service-oriented-architecture-weak-coupling]
// Кожен сервіс має мати мінімум залежностей від зовнішніх ресурсів. Це важливо для маштабування так як якщо сервіси використовують багато спільних ресурсів треба буде координувати виконання їх функцій. Також кожен повинен бути статичним, тобто не зберігати ні якої інформацій з попередніх сессій або транзакцій.
// 
// [#service-oriented-architecture-abstraction]
// Кліенти не повинні знати, як саме працює сервіс. Вони повинні знати лише які методи він надає та які параметри приймає та який результат повертає.
// 
// [#service-oriented-architecture-level-of-detail]
// Ідеально упаковивать одну бізнес-функцію в один сервіс. Розробники можуть використовувати декілька сервісів, щоб створити комбінований сервіс для виконання складних операцій.
// 
// [#service-oriented-architecture-advantages]
// Переваги SOA:
// 
// * Масштабованість
// * Гнучкість
// * Повторне використання
// * Легкість тестування
// * Легкість розгортання
// 
// [#service-oriented-architecture-disadvantages]
// Недоліки SOA:
// 
// * Складність
// * Продуктивність
// * Безпека
// * Стійкість
// * <<service-oriented-architecture-esb,ESB>>
// 
// [[service-oriented-architecture-esb]]
// Централізована точка входу в систему, яка може стати проблемою при великій кількості сервісів. ESB - є централізована сирвісна шина. Яка може мати декілька протоколів обміну даними(SOAP, AMQP, MSMQ). Якщо ESB виходить з ладу це вплаває на усі сервіси. Що може стати слабким місцем системи.
// 
// [[service-oriented-architecture-soa-problems]]SOA розв'язувати такі проблеми з <<soa-architecture,монолітньої архітектури>> як <<soa-architecture-disadvantages-scaling,масштабування>>, <<soa-architecture-disadvantages-reuse,повторне використання>>, <<soa-architecture-disadvantages-limited-fault-tolerance,обмежена відмовостійкість>>, <<soa-architecture-disadvantages-difficult-to-deploy,складність впровадження>> завдяки, тому що розбиває програмне забезпечення на незалежні сервіси які можна повторно використовувати в різних програмних системах.
//
