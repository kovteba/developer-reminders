==== Virtual Threads
= Енциклопедія Virtual Threads (Project Loom)
:toc: left
:toclevels: 3
:sectnums:
:icons: font

== 1. Проблема класичних потоків (Platform Threads)
До Java 21 кожен об'єкт `java.lang.Thread` був лише тонкою обгорткою над потоком операційної системи (**OS Thread**).

* **Пам'ять (Вага):** Кожен OS-потік резервує фіксований стек (зазвичай 1 МБ). 100,000 потоків потребують ~100 ГБ RAM лише на стеки, що робить масштабні системи занадто дорогими.
* **Context Switching:** Перемикання між потоками виконується ядром ОС (**Kernel Space**). Це вимагає збереження регістрів процесора та стану стека, що є "дорогою" операцією для CPU.
* **Блокування:** Коли Platform Thread чекає на IO, він продовжує займати свій 1 МБ пам'яті та ресурси ОС, нічого не роблячи.

== 2. Архітектура та модель M:N
Віртуальні потоки впроваджують рівень абстракції між кодом і залізом:

* **Virtual Threads (M):** Мільйони легких об'єктів у Heap (купі).
* **Carrier Threads (N):** Невелика кількість реальних OS-потоків (зазвичай рівна кількості ядер CPU), які виконують ці віртуальні потоки.
* **Планувальник:** Використовує `ForkJoinPool` у режимі FIFO.



== 3. Механіка Mounting та Unmounting (Continuation)
Це ключова «магія» Loom — здатність потоку «відкріплюватися» від носія.

1.  **Mounting:** Коли віртуальний потік готовий до роботи, планувальник призначає його на вільний **Carrier Thread**. Стан стека копіюється з купи в стек носія.
2.  **Yield (Unmounting):** Коли потік викликає блокуючу операцію (`Thread.sleep()` або `socket.read()`), він робить *yield*. JVM зберігає стан його стека (Continuation) назад у Heap.
3.  **Звільнення:** Carrier Thread стає вільним і може миттєво взяти інший віртуальний потік із черги.
4.  **Resuming:** Коли дані (або таймер) готові, планувальник знаходить будь-який вільний Carrier Thread і повертає стек віртуального потоку з купи для продовження роботи.

== 4. Механізми пробудження (Таймери та IO)

=== Робота з Thread.sleep()
1.  JVM реєструє таймер у внутрішньому сервісі `ScheduledExecutor`.
2.  Віртуальний потік відкріплюється (Unmount). Пока час не минув, він — лише пасивний об'єкт у пам'яті (0% CPU).
3.  Після сигналу таймера службовий потік JVM повертає віртуальний потік у чергу `ForkJoinPool`.

=== Робота з Network/Database IO
1.  **Реєстрація інтересу:** Замість блокування, потік реєструє свій сокет у внутрішньому **Poller** (використовує `epoll` на Linux, `kqueue` на macOS).
2.  **Очікування на рівні ОС:** Поки база даних шукає дані, Carrier Thread вільний.
3.  **Сигнал від ОС:** Коли дані приходять, ОС надсилає сигнал JVM. Poller ідентифікує віртуальний потік і ставить його в чергу на виконання.



== 5. Зв'язок із ForkJoinPool та Work Stealing
Virtual Threads базуються на перевіреному часом `ForkJoinPool`:

* **Carrier Threads** — це воркери пулу.
* **Work Stealing:** Якщо один Carrier Thread розгріб свою чергу віртуальних потоків, він "краде" задачу з черги іншого, забезпечуючи 100% завантаження ядер.
* **Різниця з минулим:** Раніше `Thread.sleep()` заблокував би воркера пулу. Тепер він викликає *yield*, дозволяючи воркеру продовжувати роботу над іншими задачами.

== 6. Аналогія: «Актор і Сцена»
* **Carrier Thread (Потік ОС)** — це реальний актор на сцені.
* **Virtual Thread** — це сценарій (роль).
* **Старий підхід:** Актор бере сценарій і, якщо там написано «чекай 5 хвилин», він тупо стоїть на сцені. Сцена зайнята.
* **Loom:** Актор кладе сценарій на край стійки, запам'ятовує рядок і бере інший сценарій. Коли "телефон задзвонить", будь-який вільний актор підбере старий сценарій і продовжить з потрібного місця.

== 7. Senior-нюанси та обмеження (Pinning)
Випадки, коли віртуальний потік не може відкріпитися від носія (**Pinning**):

1.  **synchronized блоки/методи:** Якщо всередині `synchronized` викликати IO, Carrier Thread буде заблокований.
* *Порада:* Замінюйте на `ReentrantLock`.
2.  **Native методи:** Виклики через JNI (C/C++) «приклеюють» потік до носія.
3.  **Діагностика:** Використовуйте `-Djdk.tracePinnedThreads=full` для пошуку проблемних місць.

== 8. Практична конфігурація (Spring Boot 3.2+)
Для активації всього стека (Tomcat, TaskExecutors і т.д.):

[source,properties]
----
spring.threads.virtual.enabled=true
----

== 9. Вплив на Garbage Collection (GC)
Мільйони стеков у Heap — це виклик. Рекомендується використовувати **ZGC (Z Garbage Collector)**. Він оптимізований для великої кількості об'єктів-Continuations і забезпечує паузи менше 1 мс навіть при активному перемиканні мільйонів потоків.
