[#java-collection-framework-synchronousqueue]
=== SynchronousQueue
`SynchronousQueue<E>` — це *черга нульової ємності* з пакету `java.util.concurrent`. Не зберігає жодного елемента — кожен `put()` блокується поки інший потік не викличе `take()`, і навпаки. Фактично це *точка рандеву* (rendezvous point) між producer і consumer. Не допускає `null`.

.Ієрархія наслідування
[source,subs="normal"]
----
Object
  └── AbstractCollection<E>
        └── AbstractQueue<E>
              └── <<java-collection-framework-synchronousqueue,SynchronousQueue<E>>>
----

Реалізовані інтерфейси:

* `BlockingQueue<E>` — блокуючі `put` / `take`
* `Serializable`

NOTE: `SynchronousQueue` має *ємність 0*. `peek()` завжди повертає `null`. `size()` завжди повертає `0`. `isEmpty()` завжди повертає `true`. `iterator()` повертає порожній ітератор. Це *не* звичайна черга — це механізм синхронізації.

[#java-collection-framework-synchronousqueue-constructors]
.Конструктори

.SynchronousQueue()
[source,java]
----
SynchronousQueue<String> sq = new SynchronousQueue<>();
// fair = false: порядок очікування LIFO (stack-based) — вища пропускна здатність
----

*Що робить:* створює `SynchronousQueue` у нечесному (unfair) режимі. Потоки що очікують обслуговуються у *стековому* порядку (LIFO) — покращує пропускну здатність але може спричинити starvation.

.SynchronousQueue(boolean fair)
[source,java]
----
SynchronousQueue<String> fairSQ = new SynchronousQueue<>(true);
// fair = true: порядок очікування FIFO — менша пропускна здатність, але без starvation
----

*Що робить:* при `fair=true` — потоки обслуговуються у *черговому* порядку (FIFO). При `fair=false` — у стековому (LIFO).

[[java-collection-framework-synchronousqueue-transfer-impl]]
.SynchronousQueue — TransferStack/TransferQueue, dual data structure, spin-then-park
`SynchronousQueue` реалізує два алгоритми: `TransferStack` (unfair, LIFO) та `TransferQueue` (fair, FIFO). Обидва — *dual stack/queue*: вузол може бути або DATA (producer поклав елемент) або REQUEST (consumer чекає). При зустрічі complementary вузлів → fulfill: producer передає елемент безпосередньо consumer. Spin-then-park стратегія: потік спочатку спінується кілька сотень циклів (очікуючи швидкого партнера) перед тим як `LockSupport.park()`. Все на CAS — немає lock.

[[java-collection-framework-sync-primitives-vs-queue]]
.SynchronousQueue vs CountDownLatch/CyclicBarrier/Phaser
`SynchronousQueue` — для передачі *даних* між потоками (producer/consumer). `CountDownLatch` — очікування завершення N подій (одноразове, не скидається). `CyclicBarrier` — синхронізація N потоків у точці (скидається, повторювана). `Phaser` — гнучкіший аналог Barrier з динамічною реєстрацією учасників і фазами. Жоден з barrier-примітивів не передає дані — вони лише синхронізують виконання.

[#java-collection-framework-synchronousqueue-structure]
.Внутрішній устрій
`SynchronousQueue` реалізує два алгоритми залежно від `fair`:

[source]
----
fair=false: Transferer = TransferStack (LIFO)
fair=true:  Transferer = TransferQueue (FIFO)
----

Обидва використовують *lock-free* алгоритм через CAS. Немає реального масиву чи списку — лише стек/черга *очікуючих потоків*:

[source]
----
Unfair (TransferStack) — режим стека:

Ситуація 1: producer чекає першим
  Stack: [PUT(data1)] ← top
  Consumer приходить → забирає data1, потік producer прокидається

Ситуація 2: consumer чекає першим
  Stack: [TAKE()] ← top
  Producer приходить з data → передає data consumer, обидва продовжують

Ситуація 3: fulfill (complementary operation)
  Stack: [PUT(data2), PUT(data1)] ← два producer чекають
  Consumer приходить → бере data2 (top, LIFO) — data1 чекає далі
----

.Як виглядає put/take на практиці:
[source,java]
----
SynchronousQueue<String> queue = new SynchronousQueue<>();

// Потік 1 (producer):
Thread producer = new Thread(() -> {
    try {
        System.out.println("Producer: waiting to handoff...");
        queue.put("Hello");           // ← блокується тут
        System.out.println("Producer: handoff complete!");
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
});

// Потік 2 (consumer):
Thread consumer = new Thread(() -> {
    try {
        Thread.sleep(1000);           // producer чекає 1 секунду
        System.out.println("Consumer: receiving...");
        String msg = queue.take();    // ← producer прокидається, обидва продовжують
        System.out.println("Consumer: got " + msg);
    } catch (InterruptedException e) {
        Thread.currentThread().interrupt();
    }
});

producer.start();
consumer.start();

// Вивід:
// Producer: waiting to handoff...
// Consumer: receiving...
// Producer: handoff complete!
// Consumer: got Hello
----

[#java-collection-framework-synchronousqueue-big-o-notation]
.Методи `SynchronousQueue`
[cols="2,1,4", options="header"]
|===
|`take()`                       |блокує                 |Чекає поки не прийде `put()`. Отримує елемент напряму
|`offer(E e)`                   |не блокує              |Передає елемент якщо є готовий consumer. Інакше → `false`
|`offer(E e, time, unit)`       |блокує з таймаутом     |Чекає `time`, потім → `false`
|`poll()`                       |не блокує              |Забирає якщо є готовий producer. Інакше → `null`
|`poll(time, unit)`             |блокує з таймаутом     |Чекає `time`, потім → `null`
|`peek()`                       |ніколи не блокує       |Завжди повертає `null` (немає збереженого елемента)
|`size()`                       |ніколи не блокує       |Завжди `0`
|`isEmpty()`                    |ніколи не блокує       |Завжди `true`
|`contains(Object o)`           |ніколи не блокує       |Завжди `false`
|`iterator()`                   |ніколи не блокує       |Завжди порожній ітератор
|===

NOTE: `offer(e)` без таймауту є корисним для *non-blocking handoff* — передає елемент якщо є готовий consumer, інакше відразу повертає `false` без очікування. Корисно для реалізації балансування навантаження.

.Практичні застосування

.Thread pool handoff — Executors.newCachedThreadPool()
`SynchronousQueue` використовується всередині `Executors.newCachedThreadPool()`:

[source,java]
----
// Це буквально те що відбувається в newCachedThreadPool():
ExecutorService cachedPool = new ThreadPoolExecutor(
    0,                          // corePoolSize
    Integer.MAX_VALUE,          // maximumPoolSize — необмежено потоків
    60L, TimeUnit.SECONDS,      // keepAliveTime — потік живе 60с без задачі
    new SynchronousQueue<>()    // черга нульової ємності!
);

// Як це працює:
// 1. Задача надходить → pool намагається offer() у SynchronousQueue
// 2. Якщо є вільний потік → він take()-ає задачу (прямий handoff)
// 3. Якщо вільних потоків немає → offer() повертає false → створюється новий потік
// Результат: задача завжди виконується негайно — або існуючим потоком, або новим
----

.Прямий handoff між producer і consumer потоками
[source,java]
----
SynchronousQueue<Request> handoff = new SynchronousQueue<>(true); // fair mode

// Producer потоки (HTTP workers):
for (int i = 0; i < 4; i++) {
    new Thread(() -> {
        while (true) {
            Request req = acceptRequest();
            try {
                handoff.put(req); // чекає вільного worker
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            }
        }
    }).start();
}

// Consumer потоки (business logic workers):
for (int i = 0; i < 4; i++) {
    new Thread(() -> {
        while (true) {
            try {
                Request req = handoff.take(); // чекає запиту
                processRequest(req);
            } catch (InterruptedException e) {
                Thread.currentThread().interrupt();
                break;
            }
        }
    }).start();
}
----

.SynchronousQueue vs LinkedBlockingQueue(1) vs ArrayBlockingQueue(1)
[cols="3,1,1,1", options="header"]
|===
|Зберігає елементи              |❌ ніколи          |✅ до 1                |✅ до 1
|`put` блокує якщо              |немає consumer     |є 1 елемент            |є 1 елемент
|`take` блокує якщо             |немає producer     |0 елементів            |0 елементів
|Throughput                     |✅ максимальний    |⚠️ менший              |⚠️ менший
|Lock-free                      |✅ CAS             |❌ ReentrantLock        |❌ ReentrantLock
|Використовується в             |newCachedThreadPool|загального призначення |загального призначення
|===

IMPORTANT: Використовуй `SynchronousQueue` коли потрібен *прямий handoff* між потоками без буферизації. Якщо потрібно зберегти хоча б один елемент між producer і consumer — використовуй `ArrayBlockingQueue(1)`. `SynchronousQueue` — це не черга, це *точка синхронізації*.

[[java-collection-framework-synchronousqueue-pitfalls]]
.Підводні камені

.offer() без waiting consumer
[source,java]
----
SynchronousQueue<String> q = new SynchronousQueue<>();
boolean inserted = q.offer("data"); // false! — нікого нема хто прийме
// offer() завжди false якщо нема заблокованого take() в іншому потоці
----

WARNING: `offer()` без таймауту *завжди повертає `false`* якщо в момент виклику немає потоку що очікує на `take()`. Елемент при цьому *не зберігається* — він просто відкидається. Не використовуй `offer()` якщо не впевнений що consumer вже чекає.

.Неможливі операції
[source,java]
----
SynchronousQueue<String> q = new SynchronousQueue<>();
q.peek();    // завжди null
q.size();    // завжди 0
q.isEmpty(); // завжди true
q.contains("x"); // завжди false
for (String s : q) { /* порожня ітерація — жодного елемента */ }
----

NOTE: `SynchronousQueue` не зберігає жодного елемента — тому `peek()`, `size()`, `isEmpty()`, `contains()` та `iterator()` завжди повертають "порожні" значення. Це не баг — це архітектурна особливість черги нульової ємності.

NOTE: `new SynchronousQueue<>(true)` — fair mode (FIFO між waiting threads, використовує внутрішню чергу `TransferQueue`). `new SynchronousQueue<>(false)` (default) — unfair mode (LIFO-подібний, використовує `TransferStack`, вищий throughput). Fair mode запобігає starvation окремих потоків але знижує загальну пропускну здатність.

.SynchronousQueue в ThreadPoolExecutor
[source,java]
----
// Саме SynchronousQueue використовується у Executors.newCachedThreadPool():
ThreadPoolExecutor pool = new ThreadPoolExecutor(
    0, Integer.MAX_VALUE,
    60L, TimeUnit.SECONDS,
    new SynchronousQueue<>() // кожна задача вимагає вільного потоку
);
// Якщо немає вільного потоку → offer() повертає false → створюється новий потік (до MAX)
// Якщо потік idle 60с → видаляється
// Результат: пул завжди росте під навантаженням і скорочується в спокої
----

TIP: `SynchronousQueue` у `newCachedThreadPool()` забезпечує що кожна задача негайно отримує потік — або існуючий вільний, або новостворений. Жодна задача не чекає у черзі. Це ідеально для короткочасних задач з непередбачуваним навантаженням.
