[#java-collection-framework-linkedblockingqueue]
=== LinkedBlockingQueue
`LinkedBlockingQueue` у Java — це *потокобезпечна черга на основі зв'язного списку* з пакету `java.util.concurrent`, яка реалізує інтерфейс `BlockingQueue<E>`. Може бути як *обмеженою* (bounded) так і *необмеженою* (unbounded).

Головна ідея: producer і consumer використовують *два незалежних lock* — `putLock` і `takeLock`. Завдяки цьому запис і читання можуть відбуватись *одночасно*, що дає вищу пропускну здатність порівняно з `ArrayBlockingQueue`.

* `BlockingQueue<E>` — блокуючі операції `put` / `take`
* `Queue<E>` — неблокуючі операції `offer` / `poll`
* `Collection<E>` — базові операції колекції

.Ієрархія наслідування
[source,subs="normal"]
----
Object
  └── AbstractCollection<E>
        └── AbstractQueue<E>
              └── <<java-collection-framework-linkedblockingqueue,LinkedBlockingQueue<E>>>
----

NOTE: `LinkedBlockingQueue` знаходиться у пакеті `java.util.concurrent`. На відміну від `ArrayBlockingQueue` — може бути необмеженою (default capacity = `Integer.MAX_VALUE`) і має два окремих lock для вищого throughput.

WARNING: `LinkedBlockingQueue` *не допускає `null`*. Спроба додати `null` викине `NullPointerException`.

WARNING: При використанні *необмеженої* черги (`new LinkedBlockingQueue<>()`) — `put` *ніколи не блокується*. Якщо producer швидший за consumer — черга росте необмежено і може спричинити `OutOfMemoryError`. Завжди розглядай обмежену версію для production систем.

.Конструктор LinkedBlockingQueue()
[source,java]
----
LinkedBlockingQueue<String> queue = new LinkedBlockingQueue<>();
----

*Що робить:* створює необмежену чергу з capacity = `Integer.MAX_VALUE` (~2.1 млрд). На практиці — необмежена. Голова і хвіст ініціалізуються *dummy-вузлом* (порожній `Node` без значення).

*Коли використовувати:* коли producer гарантовано не випереджає consumer і не потрібен back-pressure. Наприклад, черга задач у thread pool де consumer завжди встигає.

[source,java]
----
LinkedBlockingQueue<Runnable> taskQueue = new LinkedBlockingQueue<>();
taskQueue.put(() -> System.out.println("Task 1")); // ніколи не блокується
----

.Конструктор LinkedBlockingQueue(int capacity)
[source,java]
----
LinkedBlockingQueue<String> queue = new LinkedBlockingQueue<>(100);
----

*Що робить:* створює обмежену чергу з максимальною ємністю `capacity`. `put` блокується коли `count == capacity`. Поведінка аналогічна `ArrayBlockingQueue` але з двома lock.

*Коли використовувати:* high-throughput producer-consumer з жорстким лімітом на розмір черги. Кращий вибір ніж `ArrayBlockingQueue` якщо важлива максимальна пропускна здатність і fair режим не потрібен.

.Конструктор LinkedBlockingQueue(Collection<? extends E> c)
[source,java]
----
List<String> source = List.of("a", "b", "c");
LinkedBlockingQueue<String> queue = new LinkedBlockingQueue<>(source);
----

*Що робить:* створює необмежену чергу і послідовно додає всі елементи колекції через `add`. Ємність = `Integer.MAX_VALUE`.

*Коли використовувати:* ініціалізація черги початковим набором задач при старті системи.

NOTE: *Слабко консистентний (weakly consistent) ітератор* — на відміну від `ArrayList` ітератор `LinkedBlockingQueue` *не кидає* `ConcurrentModificationException`. Він гарантує що побачить всі елементи що існували на момент його створення, і *може* (але не зобов'язаний) побачити зміни що відбулись після.

[[java-collection-framework-linkedblockingqueue-dual-lock]]
.LinkedBlockingQueue — два lock, AtomicInteger, happens-before
`putLock` захищає запис у `last` (хвіст). `takeLock` захищає читання `head.next` (голова). `AtomicInteger count` координує між ними без спільного lock — `getAndIncrement()`/`getAndDecrement()` є атомарними. `signalNotEmpty()` викликається *після* `putLock.unlock()` бо потребує `takeLock` — захоплення вкладених lock у різних порядках = deadlock. Happens-before: `volatile write` до `count` у `put()` happens-before `volatile read` у `take()` → гарантія видимості елементу. Multi-core CPU: два lock дозволяють producer і consumer працювати одночасно на різних ядрах — throughput ~2x порівняно з `ArrayBlockingQueue` при high contention.

[[java-collection-framework-blockingqueue-threadpool]]
.ThreadPoolExecutor та BlockingQueue — lifecycle задач
`ThreadPoolExecutor` використовує `BlockingQueue` як чергу задач. При `execute(task)`: якщо `poolSize < corePoolSize` → новий потік; інакше `queue.offer(task)` — якщо `false` і `poolSize < maxPoolSize` → новий потік; інакше `RejectedExecutionHandler`. Worker-потоки: `task = queue.take()` (або `poll(keepAliveTime)` для non-core). При `shutdown()`: нові задачі відхиляються, але вже наявні у черзі *продовжують виконуватись*. `shutdownNow()` дренує чергу і повертає список незапущених задач.

.Внутрішній устрій — два lock і AtomicInteger
Це ключова відмінність від `ArrayBlockingQueue`:

[source,java]
----
// Вузол зв'язного списку
static class Node<E> {
    E item;
    Node<E> next;
    Node(E x) { item = x; }
}

private final int capacity;             // Integer.MAX_VALUE якщо unbounded
private final AtomicInteger count       // лічильник — видимий обом lock без синхронізації
    = new AtomicInteger(0);

transient Node<E> head;                 // dummy-вузол: head.next — перший реальний елемент
private transient Node<E> last;         // хвіст: останній реальний вузол

private final ReentrantLock takeLock    // lock для take, poll, peek
    = new ReentrantLock();
private final Condition notEmpty        // сигнал для заблокованих take
    = takeLock.newCondition();

private final ReentrantLock putLock     // lock для put, offer
    = new ReentrantLock();
private final Condition notFull         // сигнал для заблокованих put
    = putLock.newCondition();
----

.Dummy-вузол — навіщо він потрібен
`LinkedBlockingQueue` завжди містить *один порожній вузол на початку* (dummy / sentinel node):

[source]
----
Порожня черга:
  head → [null|next]──→ null
  last → [null|next]  (head == last == dummy)

Черга з елементами [A, B, C]:
  head → [null|next]──→ [A|next]──→ [B|next]──→ [C|next=null]
                                                      ↑
                                                     last
----

NOTE: Dummy-вузол дозволяє `putLock` і `takeLock` *не конфліктувати*. Producer завжди пише у `last`, consumer завжди читає `head.next` — це *різні вузли*, тому два lock не заважають одне одному. Без dummy-вузла при одному елементі producer і consumer торкалися б одного й того ж вузла і потребували б спільного lock.

.Як працює put(E e) — запис у хвіст
[source,java]
----
queue.put("X");
// До:    head→[∅]→[A]→[B]→[C]←last
// Після: head→[∅]→[A]→[B]→[C]→[X]←last
----

[source,java]
----
public void put(E e) throws InterruptedException {
    Objects.requireNonNull(e);
    final int c;
    final Node<E> node = new Node<E>(e);    // створюємо вузол до захоплення lock
    final ReentrantLock putLock = this.putLock;
    final AtomicInteger count = this.count;
    putLock.lockInterruptibly();            // захоплюємо ТІЛЬКИ putLock
    try {
        while (count.get() == capacity)
            notFull.await();               // звільняємо putLock і засинаємо
        enqueue(node);                     // last.next = node; last = node
        c = count.getAndIncrement();       // атомарно збільшуємо count
        if (c + 1 < capacity)
            notFull.signal();              // є ще місце — будимо наступного producer
    } finally {
        putLock.unlock();
    }
    if (c == 0)
        signalNotEmpty();                  // черга була порожня — будимо consumer
}
----

NOTE: `signalNotEmpty()` викликається *після* `putLock.unlock()`. Це важливо: сигнал до `takeLock.Condition` вимагає захоплення `takeLock`, який не можна захоплювати під `putLock` — інакше deadlock.

.Як працює take() — читання з голови
[source,java]
----
String x = queue.take();
// До:    head→[∅]→[A]→[B]→[C]←last
// Після: head→[∅|=A]→[B]→[C]←last   (старий [∅] відкидається, [A] стає новим dummy)
// Повертає: "A"
----

[source,java]
----
public E take() throws InterruptedException {
    final E x;
    final int c;
    final AtomicInteger count = this.count;
    final ReentrantLock takeLock = this.takeLock;
    takeLock.lockInterruptibly();           // захоплюємо ТІЛЬКИ takeLock
    try {
        while (count.get() == 0)
            notEmpty.await();              // звільняємо takeLock і засинаємо
        x = dequeue();                     // h = head; head = h.next; return head.item
        c = count.getAndDecrement();       // атомарно зменшуємо count
        if (c > 1)
            notEmpty.signal();             // є ще елементи — будимо наступного consumer
    } finally {
        takeLock.unlock();
    }
    if (c == capacity)
        signalNotFull();                   // черга була повна — будимо producer
    return x;
}
----

.Як dequeue() робить старий перший вузол новим dummy
[source,java]
----
private E dequeue() {
    Node<E> h = head;           // старий dummy
    Node<E> first = h.next;     // перший реальний вузол (містить "A")
    h.next = h;                 // self-link — допомагає GC (ітератор перевіряє це)
    head = first;               // перший реальний стає новим dummy
    E x = first.item;
    first.item = null;          // обнуляємо item — тепер це dummy без значення
    return x;
}
----

NOTE: Після `dequeue` старий dummy отримує `h.next = h` (self-link). Це не просто допомога GC — ітератор `LinkedBlockingQueue` перевіряє self-link щоб визначити що вузол вже видалено з черги і перестрибує до актуального `head`.

.Чому два lock дають вищий throughput
[source]
----
ArrayBlockingQueue (один lock):
  Thread-P1 →  [LOCK] put  [UNLOCK]
  Thread-C1 →          [LOCK] take [UNLOCK]   ← чекає на P1

LinkedBlockingQueue (два lock):
  Thread-P1 →  [putLock]  put  [unlock]
  Thread-C1 →  [takeLock] take [unlock]        ← не чекає на P1 !
               ↑ одночасно ↑
----

Producer захоплює `putLock` і пише у `last`. Consumer захоплює `takeLock` і читає `head.next`. Це *різні кінці списку* — вони не конфліктують. `AtomicInteger count` координує їх без спільного lock.

.Чотири групи методів

|===
|Черга *порожня* (remove)    |`remove` → `NoSuchElementException` |`poll` → `null` |`take` → блокує |`poll(t,u)` → `null`
|Черга *порожня* (examine)   |`element` → `NoSuchElementException` |`peek` → `null` | — | —
|===

.High-throughput producer-consumer
[source,java]
----
LinkedBlockingQueue<String> queue = new LinkedBlockingQueue<>(1000);

// Кілька producer потоків
ExecutorService producers = Executors.newFixedThreadPool(4);
for (int i = 0; i < 4; i++) {
    final int id = i;
    producers.submit(() -> {
        try {
            for (int j = 0; j < 250; j++) {
                queue.put("P" + id + "-Task" + j); // put і take йдуть одночасно
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    });
}

// Кілька consumer потоків
ExecutorService consumers = Executors.newFixedThreadPool(4);
for (int i = 0; i < 4; i++) {
    consumers.submit(() -> {
        try {
            while (true) {
                String task = queue.take();
                process(task);
            }
        } catch (InterruptedException e) {
            Thread.currentThread().interrupt();
        }
    });
}
----

NOTE: `Executors.newFixedThreadPool` і `Executors.newCachedThreadPool` всередині використовують `LinkedBlockingQueue` як чергу задач для `ThreadPoolExecutor`. Це найпоширеніший реальний сценарій використання.

.LinkedBlockingQueue vs ArrayBlockingQueue
[cols="3,1,1", options="header"]
|===
|Lock архітектура                   |✅ два lock (put/take незалежні) |⚠️ один спільний lock
|Throughput при high contention     |✅ вищий                   |⚠️ нижчий
|Пам'ять на елемент                 |❌ +Node overhead (~32 байт) |✅ мінімальна
|===
|CPU cache                          |❌ поганий (Node розкидані) |✅ хороший (масив)
|===
|Fair режим                         |❌ немає                   |✅ є
|===
|`size()` без lock                  |✅ AtomicInteger           |❌ потребує lock
|===
|Небезпека OOM при unbounded        |⚠️ так                    |❌ неможливо
|===
|Підходить для                      |high-throughput pipeline, thread pool |обмежений буфер, fair режим
|===
|===
|===

IMPORTANT: `LinkedBlockingQueue` є вибором за замовчуванням у `Executors.newFixedThreadPool` і `Executors.newCachedThreadPool` — це найкраще свідчення його надійності для high-throughput сценаріїв. Обирай `ArrayBlockingQueue` коли потрібен fair режим, передбачувана пам'ять або жорсткий back-pressure.

[[java-collection-framework-linkedblockingqueue-pitfalls]]
.Підводні камені

.Необмежена черга може вичерпати пам'ять
[source,java]
----
LinkedBlockingQueue<String> queue = new LinkedBlockingQueue<>(); // unbounded!
// Якщо producer швидший за consumer:
// while (true) queue.put(new Task()); // пам'ять закінчиться!
// ЗАВЖДИ вказуй capacity для production:
LinkedBlockingQueue<String> bounded = new LinkedBlockingQueue<>(10_000); // ✅
----

WARNING: `new LinkedBlockingQueue<>()` створює *необмежену* чергу з capacity = `Integer.MAX_VALUE`. Якщо producer генерує елементи швидше ніж consumer їх обробляє — черга необмежено зростає і може спричинити `OutOfMemoryError`. Для production завжди вказуй явний capacity.

NOTE: `LinkedBlockingQueue` використовує *два окремих `ReentrantLock`*: `putLock` (для запису у хвіст) і `takeLock` (для читання з голови). Завдяки цьому producer і consumer не блокують одне одного — вищий throughput ніж у `ArrayBlockingQueue` (один спільний lock). Але кожен елемент потребує окремого `Node` об'єкту — вищий memory overhead.

NOTE: `size()` у `LinkedBlockingQueue` повертає значення `AtomicInteger count` — точне і без захоплення lock (на відміну від `ArrayBlockingQueue` де `size()` захоплює lock). Але послідовність `size()` → перевірка → дія *не є атомарною* — між викликами стан черги може змінитись.

NOTE: При кожному виклику `put()` створюється новий `Node` об'єкт, а при `poll()` старий `head` стає мусором для GC. При інтенсивному put/poll-навантаженні це створює велику кількість short-lived об'єктів і збільшує тиск на garbage collector. Для GC-sensitive систем розглянь `ArrayBlockingQueue` (pre-allocated масив без Node overhead) або `ArrayDeque` (для single-threaded сценаріїв).
