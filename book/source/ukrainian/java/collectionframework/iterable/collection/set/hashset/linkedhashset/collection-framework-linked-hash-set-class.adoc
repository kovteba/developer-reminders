[#java-collection-framework-linkedhashset]
====== LinkedHashSet class
`LinkedHashSet<E>` — розширює клас `HashSet`, зберігаючи *порядок вставки* елементів завдяки використанню `LinkedHashMap` під капотом. Не допускає дублікатів. Дозволяє один `null`-елемент. Не є потокобезпечним. Операції `add`/`remove`/`contains`/`size` виконуються за O(1)* — як у `HashSet`, з невеликим overhead на підтримку двозв'язного списку.

.Ієрархія наслідування
[source,subs="normal"]
----
Object
  └── AbstractCollection<E>
        └── AbstractSet<E>
              └── <<java-collection-framework-hash-set,HashSet<E>>>
                    └── <<java-collection-framework-linkedhashset,LinkedHashSet<E>>>
----

Реалізовані інтерфейси: `Set<E>`, `Cloneable`, `Serializable`.

WARNING: `LinkedHashSet` *не є потокобезпечним*. Паралельна модифікація з кількох потоків призведе до `ConcurrentModificationException` або пошкодження структури. Для потокобезпечного Set зі збереженням порядку вставки — `Collections.synchronizedSet(new LinkedHashSet<>())` або, якщо порядок не критичний, `ConcurrentHashMap.newKeySet()`.

.Внутрішній устрій — LinkedHashMap під капотом
[source,java]
----
public class LinkedHashSet<E> extends HashSet<E> {
    // Використовує LinkedHashMap замість HashMap:
    HashSet(int initialCapacity, float loadFactor, boolean dummy) {
        map = new LinkedHashMap<>(initialCapacity, loadFactor);
    }
    // Sentinel-значення — те саме PRESENT що й у HashSet
}
----

Завдяки `LinkedHashMap`, кожен запис має додаткові поля `before` і `after` що утворюють *двозв'язний список* в порядку вставки. Саме цей список забезпечує передбачуваний порядок при ітерації.

[#java-collection-framework-linked-hash-set-constructors]
.Конструктори

.LinkedHashSet()
[source,java]
----
LinkedHashSet<String> set = new LinkedHashSet<>();
----

*Що робить:* створює порожній `LinkedHashSet` з початковою ємністю 16 і load factor 0.75.

.LinkedHashSet(Collection<? extends E> c)
[source,java]
----
List<String> list = List.of("c", "a", "b", "a", "c");
LinkedHashSet<String> set = new LinkedHashSet<>(list);
// set: [c, a, b] — дублікати видалені, порядок вставки збережений
----

*Що робить:* копіює елементи з колекції, видаляючи дублікати і зберігаючи порядок першої появи.

.LinkedHashSet(int initialCapacity)
[source,java]
----
LinkedHashSet<String> set = new LinkedHashSet<>(64);
----

.LinkedHashSet(int initialCapacity, float loadFactor)
[source,java]
----
LinkedHashSet<String> set = new LinkedHashSet<>(64, 0.5f);
----

[#java-collection-framework-linked-hash-set-methods]
.Ключові методи
[source,java]
----
LinkedHashSet<String> set = new LinkedHashSet<>();

set.add("Java");   // true  — додано, порядок: [Java]
set.add("Python"); // true  — додано, порядок: [Java, Python]
set.add("Java");   // false — дублікат, порядок НЕ змінюється: [Java, Python]
set.add(null);     // true  — один null дозволено

set.contains("Python"); // true
set.remove("Java");     // true — видалено, порядок: [Python, null]

set.size();    // 2
set.isEmpty(); // false

// Ітерація у порядку вставки — завжди передбачувано:
for (String s : set) {
    System.out.println(s); // Python, null
}
----

|===

|===

|===

NOTE: Пам'ять: +16 байт на запис порівняно з `HashSet` (поля `before` і `after` у `LinkedHashMap.Entry`). При великих множинах це може бути значним overhead.

[[java-collection-framework-linkedhashset-pitfalls]]
.Підводні камені

.LinkedHashSet — порядок вставки, не сортований
[source,java]
----
LinkedHashSet<Integer> set = new LinkedHashSet<>();
set.add(3); set.add(1); set.add(2);
System.out.println(set); // [3, 1, 2] — НЕ [1, 2, 3]!
// Якщо потрібне сортування → TreeSet
----

WARNING: `LinkedHashSet` зберігає *insertion order* (порядок вставки), а НЕ сортований порядок. Елементи 3, 1, 2 залишаться у порядку 3, 1, 2 — а не 1, 2, 3. Якщо потрібне сортування — використовуй `TreeSet`.

.Re-insertion не змінює порядок
[source,java]
----
LinkedHashSet<String> set = new LinkedHashSet<>();
set.add("a"); set.add("b"); set.add("c");
set.add("a"); // "a" вже є — порядок НЕ оновлюється!
System.out.println(set); // [a, b, c] — "a" залишилась першою
// На відміну від LinkedHashMap з access-order=true
----

NOTE: Повторний `add()` для вже наявного елемента повертає `false` і *не змінює його позицію* у порядку ітерації. Якщо потрібна семантика "перемістити в кінець при повторному доданні" — `LinkedHashMap` з `accessOrder=true` є ближчим аналогом.

.Memory overhead при великих множинах
NOTE: Якщо порядок вставки не важливий, використовуй `HashSet` замість `LinkedHashSet`. Кожен елемент у `LinkedHashSet` займає на ~16 байт більше (два додаткові посилання для двозв'язного списку). При 1 мільйоні елементів різниця — ~16 МБ.

TIP: Три правила вибору `Set`: (1) Потрібна унікальність без порядку → `HashSet`. (2) Потрібна унікальність + порядок вставки → `LinkedHashSet`. (3) Потрібна унікальність + сортування → `TreeSet`.
