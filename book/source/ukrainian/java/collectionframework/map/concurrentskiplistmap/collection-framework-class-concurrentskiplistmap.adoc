[#java-collection-framework-concurrentskiplistmap]
==== ConcurrentSkipListMap class
`ConcurrentSkipListMap<K,V>` — це *потокобезпечна сортована Map* з пакету `java.util.concurrent`, заснована на алгоритмі *skip list*. Реалізує інтерфейс `ConcurrentNavigableMap<K,V>`. Всі ключі зберігаються у *відсортованому порядку*. Очікувана складність основних операцій — O(log n). Не допускає `null`-ключів та `null`-значень. Lock-free через CAS.

.Ієрархія наслідування
[source,subs="normal"]
----
Object
  └── AbstractMap<K,V>
        └── <<java-collection-framework-concurrentskiplistmap,ConcurrentSkipListMap<K,V>>>
----

Реалізовані інтерфейси:

* `ConcurrentNavigableMap<K,V>` → `ConcurrentMap<K,V>` + `NavigableMap<K,V>`
* `SortedMap<K,V>`, `Map<K,V>`, `Cloneable`, `Serializable`

WARNING: `ConcurrentSkipListMap` *не допускає `null`*. Ні для ключів ні для значень. Спроба додати `null` викине `NullPointerException`.

NOTE: Ітератори `ConcurrentSkipListMap` є *weakly consistent* — відображають стан мапи на момент або після їх створення, але не гарантують побачення всіх змін що відбулись після створення ітератора. Не кидають `ConcurrentModificationException`.

[[java-collection-framework-concurrentskiplistmap-skiplist-theory]]
.Skip list — теорія, ймовірність p та рівні
Skip list з n елементів і ймовірністю p=0.5 per level: очікувана кількість рівнів = `log₁/ₚ(n)` = `log₂(n)`. Очікувана кількість порівнянь при пошуку = O(log n). Теоретична vs практична різниця з Red-Black Tree: обидва O(log n), але skip list простіший для concurrent реалізації (CAS per вузол замість складного балансування дерева). RB Tree краще для cache locality (компактніше). Skip list займає більше пам'яті (кілька level посилань на вузол) але простіший у concurrent сценаріях.

[[java-collection-framework-concurrentskiplistmap-cas-insert]]
.CAS-вставка та sorted order без global lock
`ConcurrentSkipListMap` вставляє через CAS loop: (1) находимо позицію O(log n); (2) `NEXT.compareAndSet(predecessor, null, newNode)` — атомарно; (3) якщо CAS провалився — retry. Sorted order підтримується бо вставка відбувається між конкретними predecessor і successor вузлами — CAS гарантує що не два вузли вставляться в одне місце. Після вставки у нижній рівень — ймовірнісне "підкидання" на вищі рівні (кожен рівень з p=0.5).

[[java-collection-framework-concurrentskiplistmap-logical-delete]]
.Logical deletion у ConcurrentSkipListMap
Видалення є двофазним: (1) Logical deletion: атомарно маркуємо вузол-наступник через CAS (`val = null` або спеціальний marker); (2) Physical deletion: відокремлюємо вузол від списку через CAS на predecessor. Логічне видалення захищає від race condition: якщо вузол фізично видаляється поки інший потік намагається вставити після нього — він побачить marker і retry. Фізична deletion може відбутися пізніше будь-яким потоком що проходить через цю позицію.

[[java-collection-framework-concurrentskiplistmap-index-structure]]
.Index та HeadIndex у ConcurrentSkipListMap
`Index<K,V>` — вузол на non-zero рівні skip list: містить `node` (посилання на базовий Node), `down` (той самий вузол рівнем нижче), `volatile right` (наступний Index того ж рівня). `HeadIndex<K,V>` наслідує Index і позначає head кожного рівня. Рівні організовані як масив HeadIndex від level 0 (всі вузли) до level k (мало вузлів). `head` — посилання на HeadIndex найвищого рівня. При resize (нові вузли піднімаються вище) — нові HeadIndex рівні додаються через CAS.

[#java-collection-framework-concurrentskiplistmap-structure]
.Внутрішній устрій — Skip List
Skip list — це ймовірнісна структура даних яка забезпечує O(log n) середню складність для пошуку, вставки і видалення через *кілька рівнів зв'язних списків*:

[source]
----
Рівень 4 (1/16 вузлів): head ──────────────────────── 50 ──────────────── tail
Рівень 3 (1/8  вузлів): head ──────────────── 30 ──── 50 ──────────────── tail
Рівень 2 (1/4  вузлів): head ──────── 20 ──── 30 ──── 50 ──── 70 ──────── tail
Рівень 1 (1/2  вузлів): head ── 10 ── 20 ──── 30 ──── 50 ──── 70 ── 90 ── tail
Рівень 0 (всі  вузли):  head ── 10 ── 20 ── 25 ─ 30 ── 50 ── 70 ─ 80 ─ 90 ── tail

Пошук 25:
  Рівень 4: head → 50 (50>25, опускаємось)
  Рівень 3: head → 30 (30>25, опускаємось)
  Рівень 2: head → 20 → 30 (30>25, опускаємось)
  Рівень 1: 20 → 30 (30>25, опускаємось)
  Рівень 0: 20 → 25 ✅  (знайдено за ~log n кроків)
----

.Як вузол skip list виглядає в коді:
[source,java]
----
static final class Node<K,V> {
    final K key;
    volatile V val;          // volatile для видимості між потоками
    volatile Node<K,V> next; // volatile для lock-free читання
}

static final class Index<K,V> {
    final Node<K,V> node;    // вузол нижнього рівня
    final Index<K,V> down;   // вузол рівня нижче
    volatile Index<K,V> right; // наступний вузол того ж рівня
}
----

.Lock-free через CAS:
[source,java]
----
// Вставка через CAS:
// 1. Знаходимо позицію O(log n)
// 2. Атомарно вставляємо через compareAndSet:
if (NEXT.compareAndSet(pred, null, newNode)) {
    // Вставка успішна
    // Інший потік міг одночасно вставляти — CAS вирішує конфлікт
}
// 3. "Підкидаємо" вузол на вищі рівні ймовірнісно (p=0.5 per level)
----

.Паралельні операції
[source,java]
----
ConcurrentSkipListMap<String, Integer> scores = new ConcurrentSkipListMap<>();

// Декілька потоків одночасно:
ExecutorService pool = Executors.newFixedThreadPool(4);

// Thread 1: додає нові рахунки
pool.submit(() -> scores.put("alice", 95));

// Thread 2: читає топ-рахунки (без lock!)
pool.submit(() -> {
    Map.Entry<String, Integer> best = scores.firstEntry();
});

// Thread 3: атомарно оновлює якщо значення стало гіршим
pool.submit(() -> scores.merge("bob", 80, Math::max));

// Thread 4: навігаційний запит
pool.submit(() -> {
    // Всі гравці з рахунком від 80 до 100:
    NavigableMap<String, Integer> topPlayers = scores.subMap("a", true, "z", true)
        .entrySet().stream()
        .filter(e -> e.getValue() >= 80)
        .collect(...);
});
----

.Навігаційні операції — практичний приклад
[source,java]
----
ConcurrentSkipListMap<Long, String> timeline = new ConcurrentSkipListMap<>();

// Timestamp → event
timeline.put(1000L, "User login");
timeline.put(2000L, "Page view");
timeline.put(3000L, "Purchase");
timeline.put(4000L, "Logout");

// Подія перед timestamp 2500:
timeline.floorEntry(2500L);   // → {2000=Page view}

// Подія після timestamp 2500:
timeline.ceilingEntry(2500L); // → {3000=Purchase}

// Всі події за останні 2 секунди:
long now = 4000L;
timeline.tailMap(now - 2000L, true); // → {3000=Purchase, 4000=Logout}

// Видалити найстаріший запис:
Map.Entry<Long, String> oldest = timeline.pollFirstEntry();
----

[#java-collection-framework-concurrentskiplistmap-pitfalls]
.Підводні камені

.O(log n) — значно повільніше O(1) ConcurrentHashMap для несортованих даних
[source,java]
----
// ConcurrentHashMap: O(1)* амортизований для put/get
// ConcurrentSkipListMap: O(log n) для put/get

// При 1 млн записів:
// ConcurrentHashMap.get("key")     → ~5-10 нс
// ConcurrentSkipListMap.get("key") → ~200-300 нс (log₂(1_000_000) ≈ 20 кроків)

// Benchmark приклад (JMH стиль):
ConcurrentHashMap<String, Integer> chm = new ConcurrentHashMap<>();
ConcurrentSkipListMap<String, Integer> cslm = new ConcurrentSkipListMap<>();

// Заповнюємо 100_000 елементів:
for (int i = 0; i < 100_000; i++) {
    String key = "key" + i;
    chm.put(key, i);
    cslm.put(key, i);
}

// get() у CHM ~3-5x швидший ніж у CSLM при великих розмірах

// ПРАВИЛО ВИБОРУ:
// Потрібна сортованість або range-запити → ConcurrentSkipListMap
// Просто concurrent Map → ConcurrentHashMap (O(1) та менше пам'яті)
----

NOTE: Використовуй `ConcurrentSkipListMap` *тільки* коли потрібна сортованість або навігаційні операції (`floorKey`, `ceilingEntry`, `headMap`, `tailMap`). Для простого concurrent доступу без сортування — `ConcurrentHashMap` є кращим вибором: O(1) операції і значно менше пам'яті.

.headMap/tailMap/subMap — живі view, а не snapshot
[source,java]
----
ConcurrentSkipListMap<Integer, String> map = new ConcurrentSkipListMap<>();
map.put(1, "one"); map.put(2, "two"); map.put(3, "three");
map.put(4, "four"); map.put(5, "five");

// subMap() повертає ЖИВИЙ view (не копію):
NavigableMap<Integer, String> sub = map.subMap(2, true, 4, true); // [2,4]

// Інший потік додає елемент у діапазон:
map.put(3, "THREE"); // sub автоматично відображає цю зміну!
System.out.println(sub); // {2=two, 3=THREE, 4=four} ← зміна видима

// Інший потік видаляє елемент з діапазону:
map.remove(3);
System.out.println(sub); // {2=two, 4=four} ← видалення теж видиме

// СЮРПРИЗ: вставка через view відображається у оригінальній map:
sub.put(3, "new-three"); // ✅ додається і в sub і в map
System.out.println(map.containsKey(3)); // true

// Спроба вставити з-поза діапазону → IllegalArgumentException:
sub.put(10, "ten"); // ❌ IllegalArgumentException! 10 > 4 (верхня межа)
sub.put(1, "one");  // ❌ IllegalArgumentException! 1 < 2 (нижня межа)

// Для snapshot — явно скопіюй:
NavigableMap<Integer, String> snapshot = new TreeMap<>(sub); // незмінна копія
----

WARNING: `headMap()`, `tailMap()`, `subMap()` та `descendingMap()` повертають *живі view* (`live views`) без копіювання даних. Зміни в оригінальній мапі автоматично відображаються у view і навпаки. Для незмінного snapshot скопіюй у `new TreeMap<>(subMap)`.

.null ключі та значення → NullPointerException
[source,java]
----
ConcurrentSkipListMap<String, Integer> map = new ConcurrentSkipListMap<>();

map.put(null, 1);    // ❌ NullPointerException!
map.put("key", null); // ❌ NullPointerException!
map.get(null);        // ❌ NullPointerException!

// ЧОМУ? Skip list сортує ключі → при порівнянні null.compareTo(x) → NPE
// Крім того, як і CHM, не допускає null для однозначності: get()==null → відсутній

// Якщо потрібна підтримка null-ключів у сортованій Map →
// Collections.synchronizedSortedMap(new TreeMap<>(Comparator.nullsFirst(...)))
// але це дасть гірший паралелізм
----

.size() є O(n) — не O(1) як у HashMap/CHM
[source,java]
----
ConcurrentSkipListMap<String, Integer> map = new ConcurrentSkipListMap<>();
for (int i = 0; i < 1_000_000; i++) {
    map.put("key" + i, i);
}

// size() в ConcurrentSkipListMap є O(n) — обходить нижній рівень skip list:
long start = System.nanoTime();
int size = map.size(); // ~мілісекунди для 1М елементів!
long elapsed = System.nanoTime() - start;
System.out.println("size()=" + size + " in " + elapsed/1_000_000 + "ms");

// Порівняй:
// HashMap.size()          → O(1), одне поле
// ConcurrentHashMap.size() → O(кількість CounterCell), але дуже швидко
// ConcurrentSkipListMap.size() → O(n) повний обхід!

// УНИКАЙ частих викликів size() у hot path!
// Якщо потрібен лічильник → ведіть його окремо через AtomicLong
AtomicLong counter = new AtomicLong();
map.put("k", 1); counter.incrementAndGet();
long count = counter.get(); // O(1) ✅
----

.Comparator має бути consistent with equals — інакше дублікати ключів
[source,java]
----
// ПАСТКА: Comparator що повертає 0 для нерівних об'єктів → "duplicate" keys
ConcurrentSkipListMap<String, Integer> map = new ConcurrentSkipListMap<>(
    (a, b) -> a.length() - b.length() // порівнює тільки довжину рядка!
);

map.put("abc", 1);  // довжина 3
map.put("def", 2);  // довжина 3 → Comparator.compare("abc","def")==0 → ПЕРЕЗАПИСУЄ!
// skip list вважає "abc" і "def" ОДНАКОВИМИ ключами (Comparator повернув 0)

System.out.println(map.size()); // 1 (!)
System.out.println(map.get("abc")); // 2 (значення перезаписано!)
System.out.println(map.get("def")); // 2

// ПРАВИЛЬНО: Comparator має повертати 0 тільки для equals-рівних об'єктів:
ConcurrentSkipListMap<String, Integer> correct = new ConcurrentSkipListMap<>(
    Comparator.comparingInt(String::length).thenComparing(Comparator.naturalOrder())
);
correct.put("abc", 1); // довжина 3, лексично "abc"
correct.put("def", 2); // довжина 3, лексично "def" → різні ключі
System.out.println(correct.size()); // 2 ✅
----

WARNING: У `ConcurrentSkipListMap` (і `TreeMap`) ключі вважаються *рівними* якщо `comparator.compare(k1, k2) == 0` — незалежно від `k1.equals(k2)`. Якщо Comparator повертає 0 для нерівних об'єктів — вони трактуються як один ключ і один перезаписує інший. Comparator *обов'язково має бути consistent with equals*.

IMPORTANT: *Обирай `ConcurrentSkipListMap` замість `Collections.synchronizedSortedMap(new TreeMap<>())`* у будь-якому багатопотоковому коді. Lock-free підхід дає набагато кращу масштабованість. Якщо сортованість не потрібна — обирай `ConcurrentHashMap` (O(1) operations).
