[#java-collection-framework-linkedhashmap]
==== LinkedHashMap class
`LinkedHashMap` — це упорядкована реалізація хеш-таблиці, яка наслідує `HashMap` і підтримує *двозв'язний список* між всіма записами. Завдяки цьому порядок ітерації є *передбачуваним*: за замовчуванням — порядок вставки (`insertion-order`), або порядок доступу (`access-order`) якщо вказати третій параметр конструктора. Не є потокобезпечною. Дозволяє `null` ключ і `null` значення.

.Ієрархія наслідування
[source,subs="normal"]
----
Object
  └── AbstractMap<K,V>
        └── <<java-collection-framework-hashmap,HashMap<K,V>>>
              └── <<java-collection-framework-linkedhashmap,LinkedHashMap<K,V>>>
----

Реалізовані інтерфейси:

* `Map<K,V>` — базовий контракт мапи
* `Cloneable` — підтримка поверхневого клонування
* `Serializable` — підтримка серіалізації

[#java-collection-framework-linkedhashmap-constructors]
.Конструктори класу `LinkedHashMap`

.LinkedHashMap()
[source,java]
----
LinkedHashMap<String, Integer> map = new LinkedHashMap<>();
----

*Що робить:* створює порожню `LinkedHashMap` з початковою ємністю 16, load factor 0.75 і режимом *insertion-order*.

*Коли використовувати:* коли потрібна Map що зберігає порядок вставки елементів.

.LinkedHashMap(int initialCapacity)
[source,java]
----
LinkedHashMap<String, Integer> map = new LinkedHashMap<>(32);
----

*Що робить:* як попередній, але з вказаною початковою ємністю.

.LinkedHashMap(int initialCapacity, float loadFactor)
[source,java]
----
LinkedHashMap<String, Integer> map = new LinkedHashMap<>(32, 0.5f);
----

*Що робить:* дозволяє задати ємність і коефіцієнт завантаження.

.LinkedHashMap(int initialCapacity, float loadFactor, boolean accessOrder)
[source,java]
----
// access-order: true = за порядком доступу (LRU), false = за порядком вставки
LinkedHashMap<String, Integer> lruCache = new LinkedHashMap<>(16, 0.75f, true);
----

*Що робить:* при `accessOrder=true` — кожен `get` або `put` переміщує запис у кінець списку (найнещодавніше використаний — останній). Основа для реалізації *LRU-кешу*.

*Коли використовувати:* для побудови кешу з витісненням найдовше невикористаних записів.

.LinkedHashMap(Map<? extends K, ? extends V> m)
[source,java]
----
Map<String, Integer> source = Map.of("a", 1, "b", 2);
LinkedHashMap<String, Integer> map = new LinkedHashMap<>(source);
----

*Що робить:* копіює всі записи з переданої мапи. Порядок ітерації відповідає порядку ітерації вихідної мапи.

WARNING: Якщо передати `null` — отримаєш `NullPointerException`.

[[java-collection-framework-linkedhashmap-entry-structure]]
.LinkedHashMap.Entry — структура та двозв'язний список
`LinkedHashMap.Entry<K,V>` наслідує `HashMap.Node<K,V>` і додає два поля: `Entry<K,V> before` (попередній у порядку вставки/доступу) та `Entry<K,V> after` (наступний). Клас підтримує sentinel-поля `head` та `tail` — голова і хвіст двозв'язного списку. Це +16 байт на кожен запис порівняно з HashMap (два додаткових reference). При ітерації LinkedHashMap обходить двозв'язний список (head.after → ... → tail), а не масив бакетів — звідси гарантований порядок і часто швидша ітерація ніж HashMap (немає пропуску порожніх бакетів). LinkedHashMap є найефективнішим способом insertion-ordered Map бо не потребує окремих структур — двозв'язний список вбудований у самі Node.

[[java-collection-framework-linkedhashmap-after-access]]
.afterNodeAccess() — переміщення у хвіст в access-order режимі
`afterNodeAccess(Node e)` викликається після кожного `get()` або `put()` на існуючий ключ в access-order режимі. Переміщує вузол у хвіст двозв'язного списку: від'єднує від поточної позиції (перезв'язує before і after сусідів), додає у tail. Це O(1) операція. `keySet()` та `values()` зберігають insertion/access order через той самий двозв'язний список. `forEach()` обходить через linked list (tail-to-head або head-to-tail), не через hash table. `containsValue()` теж обходить через linked list. `putIfAbsent()` в access-order: якщо ключ вже є — `afterNodeAccess` переміщує у хвіст, якщо новий — `afterNodeInsertion` перевіряє видалення eldest.

[[java-collection-framework-linkedhashmap-after-insertion]]
.afterNodeInsertion() — видалення eldest entry
`afterNodeInsertion(boolean evict)` викликається після кожного `put()` нового ключа. Перевіряє умову: `evict && head != null && removeEldestEntry(head)`. Якщо `removeEldestEntry()` повертає true — видаляє `head` (eldest = найдавніший у списку). За замовчуванням `removeEldestEntry()` повертає `false` (без авто-видалення).

[[java-collection-framework-linkedhashmap-removeeldest]]
.removeEldestEntry() — LRU cache hook
`removeEldestEntry(Map.Entry eldest)` — protected метод що визначає чи видаляти eldest (найдавніший) запис. Отримує eldest entry (голову двозв'язного списку). Повертає `true` → eldest видаляється автоматично після кожного put. Стандартна реалізація LRU з обмеженим розміром: `return size() > maxCapacity`. Не викликай `remove()` всередині методу — CME.

[#java-collection-framework-linkedhashmap-structure]
.Внутрішній устрій
`LinkedHashMap` додає до кожного `HashMap.Node` два додаткових поля — `before` і `after`:

[source,java]
----
static class Entry<K,V> extends HashMap.Node<K,V> {
    Entry<K,V> before;   // попередній запис у порядку вставки
    Entry<K,V> after;    // наступний запис у порядку вставки
    // ...
}
----

Крім масиву бакетів `table[]` (успадкованого від `HashMap`) є також *sentinel node* `head` і `tail` — голова і хвіст двозв'язного списку:

[source]
----
Масив бакетів (HashMap структура):
table[0] → null
table[1] → [k="b", v=2] → null
table[3] → [k="a", v=1] → null
...

Двозв'язний список (порядок вставки):
head ↔ [k="a", v=1] ↔ [k="b", v=2] ↔ tail
----

При ітерації `LinkedHashMap` проходить по двозв'язному списку (а не по масиву бакетів), тому порядок гарантований.

[#java-collection-framework-linkedhashmap-access-order]
.Access-order режим
При `accessOrder=true` кожен `get` або `put` переміщує запис у кінець списку:

[source,java]
----
LinkedHashMap<String, Integer> map = new LinkedHashMap<>(16, 0.75f, true);
map.put("a", 1);  // список: [a]
map.put("b", 2);  // список: [a, b]
map.put("c", 3);  // список: [a, b, c]
map.get("a");     // список: [b, c, a]  ← a переміщується в кінець
map.get("b");     // список: [c, a, b]  ← b переміщується в кінець
// Голова (head.after) → найдавніше використаний = "c"
----

[[java-collection-framework-linkedhashmap-lru]]
[#java-collection-framework-linkedhashmap-lru-cache]
.LRU-кеш на базі LinkedHashMap
`LinkedHashMap` надає метод `removeEldestEntry(Map.Entry)` — перевизнач його щоб автоматично витісняти найстаріший запис:

[source,java]
----
int maxSize = 3;
LinkedHashMap<String, Integer> lruCache = new LinkedHashMap<>(16, 0.75f, true) {
    @Override
    protected boolean removeEldestEntry(Map.Entry<String, Integer> eldest) {
        return size() > maxSize;
    }
};

lruCache.put("a", 1);
lruCache.put("b", 2);
lruCache.put("c", 3);
lruCache.get("a");      // [b, c, a] — b найдавніший
lruCache.put("d", 4);  // витісняє "b" → [c, a, d]
System.out.println(lruCache.keySet()); // [c, a, d]
----

NOTE: `removeEldestEntry` викликається після кожного `put`. Якщо метод повертає `true` — запис `eldest` (голова списку, тобто найдавніше використаний) автоматично видаляється. Це основа для простих LRU-кешів без додаткових залежностей.

[#java-collection-framework-linkedhashmap-big-o-notation]
.BigO notation для `LinkedHashMap`
[cols="2,1,4", options="header"]
|===
|`get(Object key)`              |O(1)*          |Пошук як у `HashMap`. При `accessOrder=true` — додатково переміщує запис у кінець списку — O(1)
|`remove(Object key)`           |O(1)*          |Видалення з `HashMap` + виключення вузла з двозв'язного списку — O(1)
|`containsKey(Object key)`      |O(1)*          |Пошук за хешем як у `HashMap`
|===
|`containsValue(Object value)`  |O(n)           |Прохід по двозв'язному списку замість масиву бакетів — але все одно O(n)
|===
|`size()`                       |O(1)           |Поле `size` успадковане від `HashMap`
|===
|`clear()`                      |O(n)           |Очищає масив бакетів і скидає `head`/`tail`
|===
|`entrySet()` / `keySet()`      |O(1)           |Повертає view. Ітерація — O(n) по двозв'язному списку
|===
|`putAll(Map m)`                |O(m)           |m — кількість елементів у переданій мапі
|===
|===
|===

NOTE: O(1)* — амортизоване O(1) при рівномірному розподілі хешів. `LinkedHashMap` трохи повільніший за `HashMap` через підтримку двозв'язного списку, але асимптотична складність однакова.

.Споживання пам'яті
Кожен запис `LinkedHashMap.Entry` займає більше пам'яті ніж `HashMap.Node`:

[source]
----
HashMap.Node:    hash (4) + key ref (4) + value ref (4) + next ref (4) = ~32 байт з заголовком об'єкта
LinkedHashMap.Entry: + before ref (4) + after ref (4)                  = ~40 байт
----

NOTE: Два додаткових посилання `before`/`after` додають *~8 байт на запис*. При мільйоні записів — ~8 МБ накладних витрат порівняно з `HashMap`.

.LinkedHashMap vs HashMap vs TreeMap
[cols="3,1,1,1", options="header"]
|===
|Швидкість get/put/remove       |✅ O(1)*               |✅ O(1)*           |⚠️ O(log n)
|`null` ключ                    |✅ 1 null ключ         |✅ 1 null ключ     |❌ (без Comparator)
|Пам'ять                        |⚠️ +8 байт на запис    |✅ мінімальна      |⚠️ +Node overhead
|===
|LRU-кеш                        |✅ ідеально            |❌                 |❌
|===
|===
|===

TIP: *Використовуй `LinkedHashMap` коли:* потрібна Map з передбачуваним порядком ітерації; будуєш простий LRU-кеш; потрібно відтворити порядок в якому елементи були додані. Для продакшн LRU-кешів з TTL і конкурентним доступом розглянь Caffeine або Guava Cache.

NOTE: Продуктивність `LinkedHashMap` дещо нижча ніж у `HashMap` через необхідність оновлювати вказівники двозв'язного списку (`before`/`after`) при кожному `put`, `get` (у access-order режимі) та `remove`. Для pure-lookup сценаріїв без вимог до порядку — `HashMap` буде швидшим.

[#java-collection-framework-linkedhashmap-pitfalls]
.Підводні камені

.Access-order + ітерація → ConcurrentModificationException
[source,java]
----
LinkedHashMap<String, Integer> map = new LinkedHashMap<>(16, 0.75f, true);
map.put("a", 1);
map.put("b", 2);
map.put("c", 3);

// ПРОБЛЕМА: get() у access-order режимі ЗМІНЮЄ порядок списку (modCount++)
// Якщо викликати get() під час ітерації → ConcurrentModificationException!
for (String key : map.keySet()) {
    map.get(key); // ❌ ConcurrentModificationException! get() модифікує список
}

// РІШЕННЯ — ітеруватися по entrySet() та читати value з entry (не через get):
for (Map.Entry<String, Integer> entry : map.entrySet()) {
    Integer val = entry.getValue(); // ✅ не викликає get() → список не змінюється
    System.out.println(entry.getKey() + "=" + val);
}

// АБО — зробити знімок ключів перед ітерацією:
List<String> keys = new ArrayList<>(map.keySet()); // копія ключів
for (String key : keys) {
    map.get(key); // ✅ ітеруємося по копії, не по живому keySet
}
----

WARNING: У `access-order` режимі *будь-який виклик `get()`* на ключ що існує в мапі переміщує відповідний запис у кінець двозв'язного списку і збільшує `modCount`. Тому ніколи не викликай `get()` через живий `keySet()` або `entrySet()` ітератор.

.LRU-кеш — повна реалізація з removeEldestEntry
[source,java]
----
public class LRUCache<K, V> extends LinkedHashMap<K, V> {
    private final int maxCapacity;

    public LRUCache(int maxCapacity) {
        // true = access-order (get переміщує запис у кінець)
        super(maxCapacity, 0.75f, true);
        this.maxCapacity = maxCapacity;
    }

    @Override
    protected boolean removeEldestEntry(Map.Entry<K, V> eldest) {
        // eldest = голова списку = найдавніше використаний запис
        return size() > maxCapacity;
    }
}

// Використання:
LRUCache<String, String> cache = new LRUCache<>(3);
cache.put("a", "alpha");   // [a]
cache.put("b", "beta");    // [a, b]
cache.put("c", "gamma");   // [a, b, c]
cache.get("a");            // [b, c, a] — 'a' переміщено у кінець
cache.put("d", "delta");   // [c, a, d] — 'b' витіснено як найдавніший
System.out.println(cache.containsKey("b")); // false — витіснено
System.out.println(cache.keySet());         // [c, a, d]
----

NOTE: `removeEldestEntry` викликається *після кожного `put`* або `putAll`. Метод отримує запис-кандидат на видалення (`eldest` = голова двозв'язного списку). Якщо метод повертає `true` — запис автоматично видаляється. Не викликай `remove()` всередині `removeEldestEntry` — це призведе до `ConcurrentModificationException`.

.Накладні витрати пам'яті — +16 байт на запис
[source,java]
----
// HashMap.Node:
//   hash (4) + key ref (8) + value ref (8) + next ref (8) + object header (16) = ~44 байт

// LinkedHashMap.Entry (наслідує Node):
//   + before ref (8) + after ref (8) = ~60 байт

// При мільйоні записів:
// HashMap:     ~44 MB
// LinkedHashMap: ~60 MB (+~16 MB накладних витрат)
// (реальні цифри залежать від JVM, padding та GC)

// Перевірка через Runtime:
long before = Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory();
Map<Long, Long> map = new LinkedHashMap<>();
for (long i = 0; i < 1_000_000; i++) map.put(i, i);
long after = Runtime.getRuntime().totalMemory() - Runtime.getRuntime().freeMemory();
System.out.println("~" + (after - before) / 1_000_000 + " bytes/entry");
// Типово ~60-80 байт/запис (з урахуванням boxing Long→Long)
----

NOTE: Два додаткових посилання `before`/`after` у `LinkedHashMap.Entry` додають *~16 байт на запис* порівняно з `HashMap.Node`. При великих мапах (мільйони записів) це відчутна різниця. Якщо порядок ітерації не потрібен — завжди обирай `HashMap`.

.null ключ у access-order режимі — переміщується в кінець
[source,java]
----
LinkedHashMap<String, Integer> map = new LinkedHashMap<>(16, 0.75f, true);
map.put(null, 0);   // [null]
map.put("a", 1);    // [null, a]
map.put("b", 2);    // [null, a, b]

// Отримання null-ключа переміщує його в кінець:
map.get(null);      // [a, b, null] — null переміщено!
System.out.println(new ArrayList<>(map.keySet())); // [a, b, null]

// У insertion-order режимі null ключ поводиться як звичайний ключ:
LinkedHashMap<String, Integer> insertOrder = new LinkedHashMap<>();
insertOrder.put(null, 0);
insertOrder.put("a", 1);
insertOrder.get(null); // порядок НЕ змінюється — [null, a]
----
