[#java-collection-framework-hashtable]
==== Hashtable class
`Hashtable` — це *legacy* реалізація інтерфейсу `Map`, яка зберігає дані у вигляді пар ключ-значення в хеш-таблиці. Клас існує з Java 1.0 (до появи Java Collections Framework) і в Java 1.2 був ретрофіттований під інтерфейс `Map`. Ключова особливість: *всі методи синхронізовані* (`synchronized`), що робить його потокобезпечним але повільним. Не допускає `null` ні для ключів ні для значень.

.Ієрархія наслідування
[source,subs="normal"]
----
Object
  └── Dictionary<K,V>          ← застарілий абстрактний клас
        └── <<java-collection-framework-hashtable,Hashtable<K,V>>>
              └── <<java-collection-framework-properties,Properties>>
----

Реалізовані інтерфейси:

* `Map<K,V>` — базовий контракт мапи
* `Cloneable` — підтримка поверхневого клонування
* `Serializable` — підтримка серіалізації

WARNING: `Hashtable` є *legacy* класом. У новому коді використовуй `ConcurrentHashMap` для потокобезпечності або `HashMap` для однопотокового коду. `Hashtable` залишено лише для зворотної сумісності.

WARNING: `Hashtable` *не допускає `null`* ні для ключів, ні для значень. Спроба додати `null` викине `NullPointerException`.

[#java-collection-framework-hashtable-constructors]
.Конструктори класу `Hashtable`

.Hashtable()
[source,java]
----
Hashtable<String, Integer> table = new Hashtable<>();
----

*Що робить:* створює порожню `Hashtable` з початковою ємністю 11 і коефіцієнтом завантаження 0.75.

*Коли використовувати:* не використовуй у новому коді — є кращі альтернативи.

.Hashtable(int initialCapacity)
[source,java]
----
Hashtable<String, Integer> table = new Hashtable<>(64);
----

*Що робить:* створює порожню `Hashtable` з вказаною початковою ємністю і коефіцієнтом завантаження 0.75. На відміну від `HashMap`, початкова ємність не округлюється до степеня двійки.

.Hashtable(int initialCapacity, float loadFactor)
[source,java]
----
Hashtable<String, Integer> table = new Hashtable<>(64, 0.5f);
----

*Що робить:* дозволяє явно задати обидва параметри — ємність і коефіцієнт завантаження.

.Hashtable(Map<? extends K, ? extends V> m)
[source,java]
----
Map<String, Integer> source = new HashMap<>();
source.put("a", 1);
Hashtable<String, Integer> table = new Hashtable<>(source);
----

*Що робить:* створює `Hashtable` і копіює всі записи з переданої мапи. Якщо `m` містить `null` ключ або значення — кидає `NullPointerException`.

[#java-collection-framework-hashtable-methods]
.Методи `Hashtable`
* `V put(K key, V value)` — додає пару ключ-значення; якщо ключ існує — перезаписує значення і повертає попереднє; метод `synchronized`;
* `V get(Object key)` — повертає значення за ключем або `null`; метод `synchronized`;
* `V remove(Object key)` — видаляє запис за ключем; метод `synchronized`;
* `boolean containsKey(Object key)` — перевіряє наявність ключа; метод `synchronized`;
* `boolean containsValue(Object value)` — перевіряє наявність значення; O(n), метод `synchronized`;
* `boolean contains(Object value)` — застарілий псевдонім `containsValue`; метод `synchronized`;
* `int size()` — повертає кількість записів; метод `synchronized`;
* `boolean isEmpty()` — перевіряє чи порожня таблиця; метод `synchronized`;
* `void clear()` — видаляє всі записи; метод `synchronized`;
* `Set<K> keySet()` — повертає Set всіх ключів;
* `Collection<V> values()` — повертає колекцію всіх значень;
* `Set<Map.Entry<K,V>> entrySet()` — повертає Set пар ключ-значення;
* `Enumeration<K> keys()` — *застарілий* метод, повертає `Enumeration` ключів; використовуй `keySet()`;
* `Enumeration<V> elements()` — *застарілий* метод, повертає `Enumeration` значень; використовуй `values()`.

[#java-collection-framework-hashtable-big-o-notation]
.BigO notation для `Hashtable`
[cols="2,1,4", options="header"]
|===
|`get(Object key)`              |O(1)*          |Пошук за хешем під `synchronized` lock
|`remove(Object key)`           |O(1)*          |Видалення за хешем під `synchronized` lock
|`containsKey(Object key)`      |O(1)*          |Пошук за хешем під `synchronized` lock
|===
|`containsValue(Object value)`  |O(n)           |Лінійний перебір всіх бакетів під `synchronized` lock
|===
|`size()`                       |O(1)           |Повертає поле `count` під lock
|===
|`clear()`                      |O(n)           |Обнуляє всі бакети під lock
|===
|===
|===

NOTE: O(1)* — амортизоване O(1) при рівномірному розподілі хешів. При поганій хеш-функції — O(n).

.Внутрішній устрій
`Hashtable` використовує *масив ланцюжків* (array of linked lists) як і `HashMap`. Але є ключова відмінність — *обчислення індексу бакета*:

[source,java]
----
// HashMap (Java 8+):
(n - 1) & hash   // бітова операція, вимагає розміру = степінь двійки

// Hashtable:
(hash & 0x7FFFFFFF) % tab.length  // залишок від ділення, будь-який розмір
----

Тому початкова ємність `Hashtable` = 11 (просте число), а не 16. Збільшення відбувається при `size > capacity * loadFactor`: новий розмір = `oldCapacity * 2 + 1`.

[source,java]
----
// Всі методи синхронізовані приблизно так:
public synchronized V get(Object key) {
    Entry<?,?> tab[] = table;
    int hash = key.hashCode();
    int index = (hash & 0x7FFFFFFF) % tab.length;
    for (Entry<?,?> e = tab[index]; e != null; e = e.next) {
        if ((e.hash == hash) && e.key.equals(key)) {
            return (V) e.value;
        }
    }
    return null;
}
----

NOTE: `synchronized` на кожному методі означає *блокування на весь об'єкт* `Hashtable`. Тисячі потоків не можуть читати одночасно — кожен чекає на звільнення lock. Це головна причина низької продуктивності порівняно з <<java-collection-framework-concurrenthashmap,`ConcurrentHashMap`>>.

.Hashtable vs HashMap vs ConcurrentHashMap
[cols="3,1,1,1", options="header"]
|===
|`null` ключ/значення           |❌ заборонено      |✅ дозволено       |❌ заборонено
|Впорядкованість                |❌ не гарантується |❌ не гарантується |❌ не гарантується
|Продуктивність                 |❌ низька (lock на весь об'єкт) |✅ висока |✅ висока
|===
|Java version                   |1.0 (legacy)       |1.2                |1.5
|===
|Рекомендовано                  |❌ legacy          |✅ однопотоковий   |✅ багатопотоковий
|===
|===
|===

[[java-collection-framework-hashtable-rehash]]
.Hashtable.rehash() vs HashMap.resize()
`Hashtable.rehash()` під час resize: новий розмір = `oldCapacity * 2 + 1` (не степінь двійки). Індекс обчислюється через `(hash & 0x7FFFFFFF) % newCapacity` (modulo division, а не бітова маска). Немає spread() — хеш не "змішується". Всі елементи перехешуються заново (дорого). У `HashMap.resize()`: новий розмір = `oldCapacity * 2` (степінь двійки). Bit trick: `hash & oldCapacity` визначає новий бакет без перерахунку хешу. Spread() вже застосований при вставці. HashMap resize значно ефективніший.

[#java-collection-framework-hashtable-pitfalls]
.Підводні камені

.Synchronized на весь об'єкт — bottleneck при багатопотоковості
[source,java]
----
Hashtable<String, Integer> table = new Hashtable<>();

// КОЖЕН метод блокує ВЕСЬ об'єкт:
// Thread A: table.get("key")     ← займає lock на весь table
// Thread B: table.get("other")   ← чекає! хоча читає інший ключ

// Порівняння з ConcurrentHashMap під навантаженням:
// Hashtable: при 16 потоках → 15 завжди чекають
// ConcurrentHashMap: при 16 потоках → різні потоки пишуть у різні бакети паралельно

// Демонстрація проблеми:
ExecutorService pool = Executors.newFixedThreadPool(16);
Hashtable<String, Integer> ht = new Hashtable<>();
ConcurrentHashMap<String, Integer> chm = new ConcurrentHashMap<>();

// Hashtable: всі 16 потоків серіалізовані через один lock
// ConcurrentHashMap: 16 потоків пишуть у 16 різних бакетів паралельно
// Результат: ConcurrentHashMap у 5-15x швидший при high contention
----

WARNING: `Hashtable` синхронізує *кожен метод* через `synchronized(this)` — один глобальний lock на весь об'єкт. При 100+ потоках всі операції серіалізуються. Використовуй `ConcurrentHashMap` — він блокує лише окремий бакет при записі і взагалі не блокується при читанні.

.null ключ або значення → NullPointerException
[source,java]
----
Hashtable<String, Integer> table = new Hashtable<>();

// NPE при null ключі:
table.put(null, 42);   // ❌ NullPointerException!

// NPE при null значенні:
table.put("key", null); // ❌ NullPointerException!

// Це відрізняє Hashtable від HashMap (що дозволяє null)
// і від ConcurrentHashMap (що також не дозволяє null, але з іншої причини)

// HashMap — для порівняння:
HashMap<String, Integer> map = new HashMap<>();
map.put(null, 42);    // ✅ один null ключ дозволено
map.put("k", null);   // ✅ null значення дозволено
----

NOTE: `Hashtable` кидає `NullPointerException` при `null` ключі або значенні тому що при обчисленні хешу викликає `key.hashCode()` без перевірки на `null`. Це проектне рішення з Java 1.0 часів — на відміну від `HashMap` де `null` обробляється явно (хеш = 0, зберігається у `table[0]`).

.Застарілий API — Enumeration замість Iterator
[source,java]
----
Hashtable<String, Integer> table = new Hashtable<>();
table.put("a", 1);
table.put("b", 2);

// ЗАСТАРІЛИЙ спосіб (Java 1.0) — Enumeration:
Enumeration<String> keys = table.keys();
while (keys.hasMoreElements()) {
    String key = keys.nextElement(); // ← застарілий API
    System.out.println(key + "=" + table.get(key));
}

Enumeration<Integer> values = table.elements();
while (values.hasMoreElements()) {
    System.out.println(values.nextElement());
}

// СУЧАСНИЙ спосіб — Iterator (через keySet/entrySet):
for (Map.Entry<String, Integer> entry : table.entrySet()) {
    System.out.println(entry.getKey() + "=" + entry.getValue());
}
// або Java 8+:
table.forEach((k, v) -> System.out.println(k + "=" + v));
----

NOTE: `keys()` та `elements()` повертають `Enumeration` — ітератор-попередник з Java 1.0 що не підтримує `remove()` під час ітерації і не є fail-fast. Завжди використовуй `keySet()`, `values()`, `entrySet()` або `forEach()`.

.Хеш-формула — потенційно більше колізій
[source,java]
----
// HashMap (Java 8+) — spread хеш:
static final int hash(Object key) {
    int h = key.hashCode();
    return h ^ (h >>> 16); // змішує старші та молодші біти → менше колізій
}
// Індекс: (n-1) & hash  ← бітова маска, n завжди степінь 2

// Hashtable — прямий хеш:
int hash = key.hashCode();
int index = (hash & 0x7FFFFFFF) % tab.length; // modulo division
// Немає розмішування бітів → при поганій hashCode() більше колізій
// tab.length НЕ є степенем 2 (11, 23, 47...) → modulo замість битової маски

// При одній і тій же hashCode() функції ключів:
// HashMap: рівномірніший розподіл завдяки XOR spread
// Hashtable: потенційно більше колізій у деяких бакетах
----

.Properties як підклас Hashtable — несподіваний instanceof
[source,java]
----
Properties props = new Properties();
props.setProperty("key", "value");

// Properties — підклас Hashtable!
System.out.println(props instanceof Hashtable); // true
System.out.println(props instanceof Map);        // true

// Це дозволяє передати Properties туди де очікується Hashtable або Map:
void processMap(Map<Object, Object> map) { ... }
processMap(props); // ✅ компілюється — Properties є Map<Object,Object>

// АЛЕ: передача як Map<String,String> не компілюється (type erasure):
// Map<String,String> m = props; // ❌ compile error — Hashtable<Object,Object>

// Ця ієрархія вважається помилкою дизайну — Properties "є" Hashtable тільки з
// точки зору наслідування, але концептуально це не так (Properties має власну
// семантику та defaults-ланцюжок).
Hashtable<Object, Object> legacy = props; // ✅ компілюється
----

IMPORTANT: *Ніколи не використовуй `Hashtable` у новому коді.* Для потокобезпечної мапи — `<<java-collection-framework-concurrenthashmap,ConcurrentHashMap>>`. Для однопотокового коду — `<<java-collection-framework-hashmap,HashMap>>`. Якщо потрібна `null`-підтримка під lock — `Collections.synchronizedMap(new HashMap<>())`.
