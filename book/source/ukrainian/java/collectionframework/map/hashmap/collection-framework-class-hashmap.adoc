[#java-collection-framework-hashmap]
==== HashMap class
`HashMap<K,V>` — це *хеш-таблиця*, що реалізує інтерфейс `Map<K,V>` і наслідує `AbstractMap<K,V>`. Найбільш широко вживана реалізація `Map`. Зберігає пари ключ-значення *без гарантованого порядку*. Допускає один `null`-ключ і довільну кількість `null`-значень. В середньому O(1) для `get`/`put`/`remove`/`containsKey`, але без thread-safety гарантій.

.Ієрархія наслідування
[source,subs="normal"]
----
Object
  └── AbstractMap<K,V>
        └── <<java-collection-framework-hashmap,HashMap<K,V>>>
              └── <<java-collection-framework-linkedhashmap,LinkedHashMap<K,V>>>
----

Реалізовані інтерфейси: `Map<K,V>`, `Cloneable`, `Serializable`.

WARNING: `HashMap` *не є потокобезпечним*. Паралельна запис з будь-яких двох потоків може пошкодити внутрішню структуру (нескінченна петля в Java 7 при resize, втрата елементів у Java 8+). В багатопотоковому коді використовуй `ConcurrentHashMap`.

NOTE: `HashMap` дозволяє *рівно один* `null`-ключ (зберігається в `table[0]`, хеш = 0) та будь-яку кількість `null`-значень. `Hashtable` та `ConcurrentHashMap` `null` не приймають взагалі.

[[java-collection-framework-hashmap-power-of-two]]
.Чому capacity завжди степінь двійки — bit trick `(n-1) & hash`
`HashMap` вимагає щоб розмір внутрішньої таблиці (`table.length`) завжди був степенем двійки. Це дозволяє замінити повільну операцію `hash % n` на ефективну бітову маску `(n-1) & hash`. Оскільки `n` — степінь двійки, `(n-1)` має вигляд `0b0...01111`, що є природньою маскою для молодших бітів хешу. Якщо передати `initialCapacity=100` — реальний розмір буде 128 (наступний степінь двійки).

[[java-collection-framework-hashmap-loadfactor-analysis]]
.Load factor та memory-performance trade-off
Load factor (default 0.75) визначає коли відбувається resize: `threshold = capacity * loadFactor`. При `loadFactor=0.75` resize відбувається коли таблиця заповнена на 75%. Менший load factor → менше колізій але більше пам'яті (більше порожніх бакетів). Більший → менше пам'яті але більше колізій. Значення 0.75 обрано як компроміс між пам'яттю і часом доступу. Для N очікуваних елементів: `initialCapacity = (int)(N / 0.75f + 1.0f)`.

[[java-collection-framework-hashmap-lazy-table]]
.Lazy initialization внутрішнього масиву
Порожній `HashMap` (`new HashMap<>()`) не створює внутрішній масив `table[]` одразу. Масив ініціалізується лазово при першому `put()`. `entrySet().iterator()` на порожньому HashMap повертає порожній ітератор без створення таблиці. Це заощаджує пам'ять коли HashMap створюється але ніколи не заповнюється.

[#java-collection-framework-hash-map-constructors]
.Конструктори

.HashMap()
[source,java]
----
HashMap<String, Integer> map = new HashMap<>();
----

*Що робить:* створює порожній `HashMap` з *початковою ємністю 16* і *load factor 0.75*. Масив бакетів не ініціалізується одразу — він створюється при першому `put` (lazy initialization). Threshold для першого resize = 16 * 0.75 = 12.

[[java-collection-framework-hash-map-buckets-default-amount]]
*Коли використовувати:* більшість випадків. Якщо заздалегідь невідомо скільки буде елементів.

.HashMap(int initialCapacity)
[source,java]
----
// Планується зберігати ~100 елементів:
// Правильна формула: expectedSize / loadFactor + 1
// 100 / 0.75 + 1 ≈ 135 → округлюємо до 256 (наступний степінь 2)
HashMap<String, Integer> map = new HashMap<>(256);

// Або з Java Guava:
HashMap<String, Integer> map2 = Maps.newHashMapWithExpectedSize(100);
----

*Що робить:* встановлює початкову ємність. Реальна ємність завжди є *степенем двійки* — якщо передати 100, буде 128. Це дозволяє використовувати `(n-1) & hash` замість дорогої операції `%`.

*Коли використовувати:* коли кількість елементів відома заздалегідь — уникає дорогих операцій resize/rehash.

TIP: Для `n` очікуваних елементів передавай `initialCapacity = n / 0.75 + 1` (або `(int)(n / 0.75f + 1.0f)`), щоб гарантовано уникнути хоча б одного resize.

.HashMap(int initialCapacity, float loadFactor)
[source,java]
----
// Load factor 0.5 — менше колізій, більше пам'яті:
HashMap<String, Integer> lowCollisions = new HashMap<>(32, 0.5f);

// Load factor 0.9 — більше колізій, менше пам'яті:
HashMap<String, Integer> memoryEfficient = new HashMap<>(32, 0.9f);
----

*Що робить:* налаштовує trade-off між пам'яттю і колізіями. При `loadFactor < 0.75` — більше пам'яті але менше колізій (швидше). При `loadFactor > 0.75` — менше пам'яті але більше колізій (повільніше).

*Коли використовувати:* якщо `get`/`containsKey` критичні за швидкістю → знижуй `loadFactor`. Якщо критична пам'ять → підвищуй `loadFactor`. Default 0.75 є добре збалансованим.

.HashMap(Map<? extends K, ? extends V> m)
[source,java]
----
Map<String, Integer> source = Map.of("a", 1, "b", 2, "c", 3);
HashMap<String, Integer> map = new HashMap<>(source);
----

*Що робить:* створює `HashMap` і додає всі записи з `m`. Початкова ємність = `max(16, m.size() / 0.75 + 1)` округлена до степеня двійки.

[[java-collection-framework-hashmap-node-memory]]
.HashMap.Node vs HashMap.TreeNode — пам'ять та структура
`HashMap.Node` — базовий вузол для ланцюжка: `hash` (4 байт) + `key` ref (8) + `value` ref (8) + `next` ref (8) + object header (16) ≈ 48 байт. `HashMap.TreeNode` наслідує від `LinkedHashMap.Entry` (яка наслідує `Node`) і додає: `parent`, `left`, `right`, `prev` references + `boolean red` ≈ 96 байт (вдвічі більше). `TreeNode` наслідує від `Node` а не є окремою ієрархією щоб treeified і untreeified бакети могли зберігатися в одному масиві `Node[]` без зміни типу масиву.

[#java-collection-framework-hashmap-internal-structure]
.Внутрішній устрій — три стани бакета

[[java-collection-framework-hash-map-methods-structure]]
Всередині `HashMap` — масив `Node<K,V>[] table`. Кожен елемент масиву (бакет) може перебувати в одному з трьох станів:

[source]
----
1. null         → бакет порожній

2. Node<K,V>    → ланцюжок (linked list), якщо ≤ 8 елементів:
   table[i] → [k1,v1,hash1,next] → [k2,v2,hash2,next] → null

3. TreeNode<K,V> → червоно-чорне дерево, якщо > 8 елементів і table.length ≥ 64
----

.Структура Node (Java 8+)
[source,java]
----
static class Node<K,V> implements Map.Entry<K,V> {
    final int hash;    // кешований хеш ключа — обчислюється при put, не перераховується
    final K key;       // ключ immutable після вставки
    V value;           // значення може змінюватись через put/replace
    Node<K,V> next;    // наступний вузол у ланцюжку (при колізії)
}
----

.Структура TreeNode (Java 8+, для великих бакетів)
[source,java]
----
static final class TreeNode<K,V> extends LinkedHashMap.Entry<K,V> {
    TreeNode<K,V> parent;  // батько в дереві
    TreeNode<K,V> left;    // ліве поддерево
    TreeNode<K,V> right;   // праве піддерево
    TreeNode<K,V> prev;    // для untreeify
    boolean red;           // колір вузла (Red-Black Tree)
}
----

NOTE: `TreeNode` займає *приблизно вдвічі більше пам'яті* ніж `Node` (близько 48 байт vs 32 байти). Treeification відбувається тільки коли є реальна проблема з колізіями — кількість вузлів у бакеті > `TREEIFY_THRESHOLD = 8` *і* загальний розмір таблиці ≥ `MIN_TREEIFY_CAPACITY = 64`.

[[java-collection-framework-hashmap-treeify-stats]]
.Статистичне обґрунтування TREEIFY_THRESHOLD=8 (Poisson distribution)
За умови рівномірного розподілу хешів ймовірність k колізій у одному бакеті підпорядковується розподілу Пуассона з λ=0.5 (при load factor 0.75): P(k≥8) ≈ 0.00000006 (6 на 100 мільйонів). Число 8 обрано як threshold де ймовірність природньої колізії настільки мала що її наявність свідчить про навмисну атаку або дефектну hashCode(). `UNTREEIFY_THRESHOLD=6` (а не 8) — гістерезис щоб уникнути постійного treeify/untreeify при видаленні/вставці елементів на межі 8. Якщо `table.length < MIN_TREEIFY_CAPACITY=64` і бакет > 8 елементів — виконується `resize()` замість treeify (більше бакетів краще розподілять ключі).

.Ключові константи
[source,java]
----
static final int DEFAULT_INITIAL_CAPACITY = 16;    // стартова ємність
static final int MAXIMUM_CAPACITY = 1 << 30;       // максимум (~1 млрд)
static final float DEFAULT_LOAD_FACTOR = 0.75f;    // load factor за замовчуванням
static final int TREEIFY_THRESHOLD = 8;            // поріг перетворення в дерево
static final int UNTREEIFY_THRESHOLD = 6;          // поріг повернення до ланцюжка
static final int MIN_TREEIFY_CAPACITY = 64;        // мінімальний розмір для treeify
----

[[java-collection-framework-hashmap-spread-detail]]
.Як spread() (функція hash()) покращує розподіл хешів
Без `spread()` при маленькій таблиці (n=16) оператор `(n-1) & hash` використовує лише 4 молодших біти хешу. Якщо об'єкти мають hashCode що відрізняються тільки у старших бітах — всі вони потраплять в один бакет. XOR верхніх 16 біт з нижніми 16 (`h ^ (h >>> 16)`) "змішує" старші біти у молодші, забезпечуючи рівномірніший розподіл навіть при маленьких таблицях.

[#java-collection-framework-hashmap-hash]
.Хеш-функція — spread()
При виклику `put(key, value)` або `get(key)` перший крок — обчислення хешу:

[source,java]
----
static final int hash(Object key) {
    int h;
    return (key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16);
}
----

Чому не просто `key.hashCode()`? Тому що `(n-1) & hash` при маленькій таблиці (n=16) використовує тільки *4 молодших біти* хешу. Якщо старші біти у різних ключів однакові — всі вони потраплять в один бакет. XOR з `h >>> 16` «змішує» старші біти з молодшими — рівномірніший розподіл при будь-якому `n`.

[source]
----
key.hashCode()    = 10101010_11001100_01010101_10101010
h >>> 16          = 00000000_00000000_10101010_11001100
XOR результат     = 10101010_11001100_11111111_01100110
                                       ↑ молодші біти стали «кращими»
(n-1) & hash [n=16] використовує лише ці 4 біти ─────┘
----

[[java-collection-framework-hashmap-tree-vs-list-collision]]
.Обробка колізій у treeified bucket vs linked list bucket
У ланцюжку (linked list) нові вузли додаються у кінець, пошук — лінійний O(k). У treeified бакеті (Red-Black Tree) пошук і вставка — O(log k). При вставці у дерево без `Comparable` для ключів `putVal()` використовує `System.identityHashCode()` як tie-breaker для визначення позиції у дереві, що забезпечує детерміновану структуру навіть без `Comparable`.

[[java-collection-framework-hashmap-key-comparison]]
.Послідовність порівняння ключів у putVal()
`HashMap.putVal()` порівнює ключі у такому порядку: (1) `hash == existingHash` — швидка перевірка (ціла числа); (2) `key == existingKey` — identity check (`==`); (3) `key.equals(existingKey)` — логічна рівність. Перевірка `hash == hash` перед `equals()` дає значне прискорення — хеш порівнюється як ціле число за одну інструкцію процесора. `equals()` може бути дорогим (особливо для String або складних об'єктів). Два ключі з однаковим hash І `equals() == true` — це той самий ключ, значення перезаписується.

[#java-collection-framework-hash-map-class-put-method]
.Як працює put(key, value)

[source,java]
----
map.put("name", "Alice");
----

[source]
----
1. hash = spread("name".hashCode())                    → int hash

2. i = (table.length - 1) & hash                      → індекс бакету
   Чому (n-1)&hash а не hash%n? Бо n — завжди степінь 2,
   тому (n-1) = 0b...01111 — маска, яка ефективніша ніж %

3. table[i] == null?
   → new Node(hash, key, value, null)                 → O(1), кінець

4. table[i] != null:
   a. key == existingKey || key.equals(existingKey)?
      → перезаписуємо value                           → O(1)
   b. Це TreeNode? → вставка в дерево                 → O(log n)
   c. Ланцюжок:    → йдемо по next...next до кінця
      при збігу ключа — перезаписуємо
      якщо не знайдено — додаємо в кінець             → O(k), k=довжина ланцюжка

5. if (++size > threshold) resize()                    → можливо O(n)
----

[[java-collection-framework-hash-map-put-return]]

.Повернене значення put()
[source,java]
----
HashMap<String, Integer> map = new HashMap<>();

// put повертає ПОПЕРЕДНЄ значення або null якщо ключа не було:
Integer prev1 = map.put("A", 1);  // null (ключ "A" новий)
Integer prev2 = map.put("A", 2);  // 1    (повернув стар значення)
Integer prev3 = map.put("B", null); // null (значення null — ключ новий)
Integer prev4 = map.put("B", 3);  // null! (попереднє значення було null)
// Увага: не можна відрізнити "ключа не було" від "значення було null" по поверненому null!
// Для перевірки наявності ключа: map.containsKey("B")
----

.Як працює get(key)
[source,java]
----
map.get("name");
----

[source]
----
1. hash = spread("name".hashCode())
2. i = (n-1) & hash
3. e = table[i]
4. e == null?     → return null
5. e.hash == hash && (e.key == key || key.equals(e.key))? → return e.value   (O(1))
6. e instanceof TreeNode? → tree.find(hash, key)                               (O(log n))
7. Ланцюжок: while (e.next != null) → перевіряємо кожен вузол                 (O(k))
----

NOTE: `get()` завжди *без lock* і не є atomic-safe у багатопотоковому середовищі. При паралельній записі `get()` може повернути застарілі або пошкоджені дані.

[[java-collection-framework-hashmap-resize-treenode]]
.TreeNode під час resize() — bit trick для розподілу по новим бакетам
Під час resize() `TreeNode` визначає свій новий бакет через той самий bit trick що і звичайний `Node`: `hash & oldCapacity`. Якщо результат 0 — вузол лишається на старому індексі. Якщо ненульовий — переходить на `oldIndex + oldCapacity`. Хеш вже кешований у полі `Node.hash` — перерахунок не потрібен. Після переміщення `TreeNode` перевіряє розмір: якщо ≤ `UNTREEIFY_THRESHOLD=6` — перетворюється назад у ланцюжок (`untreeify()`).

[#java-collection-framework-hashmap-resize]
.Resize / Rehashing

[[java-collection-framework-hash-map-buckets-amount]]

Коли `size > threshold` (де `threshold = capacity * loadFactor`) викликається `resize()`:

[source]
----
1. Новий масив: newCapacity = oldCapacity * 2
2. newThreshold = newCapacity * loadFactor
3. Для кожного бакету — перерозподіл елементів:
   Оскільки capacity завжди степінь 2, новий індекс = або старий індекс,
   або старий індекс + oldCapacity:

   Елемент із хешем hash:
   Старий індекс: (oldCap - 1) & hash
   Новий bit:     hash & oldCap (старший нový біт маски)
   → 0: елемент лишається на тому ж місці
   → 1: елемент переходить на oldIndex + oldCap
----

[source,java]
----
// Приклад: oldCap=16, newCap=32
// hash=0b...10110 (22)
// Старий індекс: (16-1) & 22 = 0b01111 & 0b10110 = 0b00110 = 6
// Новий bit:      16 & 22    = 0b10000 & 0b10110 = 0b10000 = 16 (ненульовий)
// Новий індекс:   6 + 16 = 22
----

TIP: Завдяки цьому трюку `resize()` не потребує перерахунку хешу — хеш вже кешований у `Node.hash`. Це в *2 рази* швидше ніж у `Hashtable` де `hash % newCapacity` перераховується для кожного елементу.

.Treeification — перетворення ланцюжка в дерево

Коли кількість елементів у одному бакеті перевищує `TREEIFY_THRESHOLD = 8`:

[source]
----
Умова 1: binCount > 8
Умова 2: table.length >= 64 (MIN_TREEIFY_CAPACITY)

Якщо умова 1 але НЕ умова 2 → resize() замість treeify
(маленька таблиця → краще збільшити кількість бакетів)

Якщо обидві умови → ланцюжок перетворюється в TreeNode
(збалансоване червоно-чорне дерево → O(log n) замість O(n))

Зворотне перетворення (untreeify):
Коли після remove кількість вузлів ≤ UNTREEIFY_THRESHOLD = 6
→ дерево знову стає ланцюжком
----

[[java-collection-framework-hashmap-amortized]]
.Амортизована складність HashMap
O(1) гарантований (без амортизації): `size()`, `isEmpty()`, `containsKey()` (за умови O(1) hashCode і equals), `get()` для першого вузла бакета. Amortized O(1): `put()` — в середньому O(1) але рідко O(n) при resize. Амортизація не застосовується при: послідовних insertions з подальшим resize (кожен n-й put — O(n)); поганій hashCode() де всі елементи у одному бакеті → O(n) або O(log n) після treeify.

|`forEach(action)`              |O(n)           |Прохід по всіх бакетах і вузлах
|`merge(k, v, remapFn)`         |O(1)*          |Дуже зручно для підрахунку (word frequency)
|`computeIfAbsent(k, fn)`       |O(1)*          |Вставляє якщо ключ відсутній. Корисно для кешування
|`getOrDefault(k, default)`     |O(1)*          |Зручна альтернатива null-check
|===

NOTE: O(1)* — *амортизоване O(1)* за умови рівномірного розподілу хешів. [[java-collection-framework-hash-map-big-o-get-element]]При деградованій хеш-функції (наприклад `hashCode()` завжди повертає 0) — всі елементи в одному бакеті → O(n) для ланцюжка або O(log n) після treeification.

[[java-collection-framework-hashmap-worst-case-hashcode]]
.Найгірший випадок hashCode() та деградація HashMap
Якщо `hashCode()` завжди повертає 0 — всі ключі потрапляють у бакет 0. До Java 8: деградація до O(n) (один довгий ланцюжок). Java 8+: після 8 елементів — treeification → O(log n). Але навіть O(log n) є катастрофічним при хеш-флуд атаках. Найгірша послідовність до Java 8: ключі що дають однаковий hash модуло capacity (наприклад рядки "Aa", "BB", "Ca" мають однаковий Java hashCode). Java 8 вирішує це через treeification, але не через hash diversification.

[[java-collection-framework-hashmap-integer-edge-cases]]
.Integer.MIN_VALUE як ключ HashMap — edge case
`Integer.MIN_VALUE` є валідним ключем `HashMap`. `Integer.MIN_VALUE.hashCode()` = `Integer.MIN_VALUE` (-2147483648). `spread()` застосовується нормально: `h ^ (h >>> 16)` = `-2147483648 ^ 32768` = результат > 0 в `int` представленні. `(n-1) & hash` коректно дає індекс. Жодного edge case немає — `Integer` є immutable, hashCode детермінований.

[#java-collection-framework-hashmap-pitfalls]
.Підводні камені та нюанси

[[java-collection-framework-hashmap-java7-infinite-loop]]
.HashMap Java 7 — нескінченна петля при concurrent resize
У Java 7 `HashMap.transfer()` під час resize переміщував елементи з старого масиву у новий через reversed linked list — новий елемент вставлявся у ГОЛОВУ ланцюжка. При паралельній записи двох потоків міг виникнути циклічний список: потік A та потік B обидва виконують resize, A переставляє вузли у зворотному порядку, B бачить вже переставлені вузли → `next` вузла вказує назад → нескінченний цикл у `get()`. Java 8 виправив це: елементи вставляються у тому ж порядку (tail insertion) — але HashMap все ще не є thread-safe. Під concurrent puts у Java 8 можлива втрата елементів (не нескінченний цикл).

[[java-collection-framework-hashmap-read-thread-safety]]
.HashMap та JMM — видимість між потоками
`HashMap` не є thread-safe навіть для read-only операцій бо внутрішні поля (`table`, `size`, `modCount`) не є `volatile`. JMM не гарантує visibility без happens-before. Потік B може бачити застарілий `table` якщо потік A виконав `put()` без синхронізації. `resize()` є особливо небезпечним — під час resize `table` замінюється новим масивом; потік що читає `table` посередині resize може бачити неповний стан.

[[java-collection-framework-hashmap-null-key-impl]]
.HashMap null-ключ — реалізація
`null`-ключ зберігається у `table[0]` (бакет 0) з хешем 0. Функція `hash(null)` повертає 0. `(n-1) & 0 = 0`. Тому null-ключ завжди в нульовому бакеті. `putVal` перевіряє `(key == null) ? 0 : (h = key.hashCode()) ^ (h >>> 16)`. Дозволений рівно один null-ключ — повторний `put(null, newVal)` перезаписує значення.

[[java-collection-framework-hashmap-hook-methods]]
.HashMap hook-методи afterNodeAccess та afterNodeInsertion
`HashMap` визначає порожні protected методи `afterNodeAccess(Node e)`, `afterNodeInsertion(boolean evict)`, `afterNodeRemoval(Node e)`. Вони викликаються при кожному відповідному internal event але нічого не роблять у `HashMap`. `LinkedHashMap` перевизначає їх щоб підтримувати двозв'язний список (переміщення у хвіст при access-order, видалення eldest entry). Цей шаблон — Template Method Pattern — дозволяє `LinkedHashMap` розширити поведінку без дублювання логіки `HashMap`.

[[java-collection-framework-hashmap-live-views]]
.Live views у HashMap — entrySet, keySet, values
`entrySet()`, `keySet()`, `values()` повертають *live view* — view-об'єкт що делегує всі операції назад до `HashMap`. Вони не є snapshot (копією). Зміни у `HashMap` відразу видимі через view і навпаки. `values()` повертає `Collection` (не `Set`) тому що значення можуть дублюватися — `Set` вимагає унікальності. `keySet()` повертає `Set` бо ключі унікальні. Ітерація по views проходить по масиву бакетів: спочатку non-null бакети, потім їх ланцюжки.

[[java-collection-framework-hashmap-serialization]]
.Серіалізація HashMap — чому table є transient
`table` є `transient` бо хеші ключів залежать від JVM (hashCode() не детермінований між JVM сесіями для деяких типів, наприклад Object). Під час серіалізації записуються лише ключі і значення (через custom `writeObject`), але не сама таблиця. При десеріалізації (`readObject`) таблиця відбудовується заново з нуля — всі ключі рехешуються. Це гарантує коректну структуру після десеріалізації навіть якщо JVM дає різні hashCode для об'єктів.

[[java-collection-framework-hashmap-modcount]]
.modCount та fail-fast механізм HashMap
`modCount` — лічильник структурних модифікацій (put нового ключа, remove, clear, resize, putAll). Iterator зберігає `expectedModCount = map.modCount` при створенні. При кожному `next()` перевіряє `modCount == expectedModCount`. Якщо не рівні — кидає `ConcurrentModificationException`. Структурні модифікації: `put()` (тільки якщо новий ключ), `remove()`, `clear()`, `resize()`, `putAll()`. НЕ є структурними: `put()` для вже існуючого ключа (перезапис значення), `get()`, `containsKey()`, `size()`, `replaceAll()` якщо ключ вже є (але `replaceAll` насправді не збільшує modCount).

[[java-collection-framework-hashmap-compute-methods]]
.HashMap compute методи — compute, computeIfAbsent, computeIfPresent, merge
`computeIfAbsent(k, fn)` — викликає `fn(k)` ТІЛЬКИ якщо ключ відсутній або має значення null, вставляє результат. Рекурсивний виклик `computeIfAbsent(sameKey, fn)` з того ж потоку в тій же `fn` — небезпечно: у Java 8 може повернути null або зависнути, у Java 9+ поведінка документована як UB. `computeIfPresent(k, fn)` — виклик `fn(k, oldVal)` тільки якщо ключ є і значення non-null. `compute(k, fn)` — завжди виклик, може створити або видалити (якщо fn повертає null). `merge(k, v, fn)` — якщо ключ відсутній → вставляє `v`; якщо є → виклик `fn(oldVal, v)`; якщо `fn` повертає null → видаляє ключ.

[[java-collection-framework-hashmap-foreach-vs-iterator]]
.HashMap.forEach() vs entrySet().iterator()
`forEach(BiConsumer)` (Java 8+) ітерує по масиву бакетів напряму без створення `Entry` view-об'єктів для кожного елемента. `entrySet().iterator()` створює `EntryIterator` і при кожному `next()` повертає поточний `Node` як `Entry`. `forEach` зазвичай трохи швидший бо уникає проміжних об'єктів та перевірок. Обидва перевіряють `modCount`. `forEach` не підтримує `remove()` під час ітерації — для цього потрібен `entrySet().removeIf()` або `iterator.remove()`.

[[java-collection-framework-hashmap-putall]]
.HashMap.putAll() vs послідовні put()
`putAll(Map m)` може бути ефективнішим: перед вставкою перевіряє розмір вхідної мапи і якщо `m.size() > threshold` — відразу виконує resize до потрібного розміру (один resize замість кількох). Це запобігає каскадним resize. Але `putAll` НЕ атомарний — інші потоки можуть бачити частково вставлений стан.

[[java-collection-framework-hashmap-getordefault]]
.HashMap.getOrDefault() — зручність, не атомарність
`getOrDefault(key, defaultValue)` повертає значення для ключа або `defaultValue` якщо ключ відсутній. НЕ є atomic — між `get` і поверненням дефолту інший потік може вставити ключ (у багатопотоковому коді). Для потокобезпечного варіанту використовуй `ConcurrentHashMap.getOrDefault()` або `computeIfAbsent`.

[[java-collection-framework-hashmap-remove-conditional]]
.HashMap.remove(key, value) — умовне видалення
`remove(Object key, Object value)` видаляє запис тільки якщо і ключ і значення збігаються (`equals()`). Повертає `boolean`. НЕ є atomic в `HashMap` — одна операція але без lock. У `ConcurrentHashMap` `remove(k,v)` є атомарним (під lock на бакет).

[[java-collection-framework-hashmap-equals-contract]]
.HashMap та порушений equals() контракт
Якщо `equals()` не рефлексивний (`a.equals(a) == false`): `putVal()` перевіряє `key == existingKey` (`==`, true для одного об'єкта) перед `equals()`. Тому identity check `==` захищає від нерефлексивного `equals`. Але якщо ключ не той самий об'єкт — `equals(a, a)` повернув би false → HashMap вважатиме їх різними ключами → дублювання. Завжди дотримуйся контракту `equals()`.

[[java-collection-framework-hashmap-clone]]
.HashMap.clone() — shallow copy
`clone()` створює новий `HashMap` того ж розміру і копіює всі entries. Але це shallow copy: ключі і значення — ті самі об'єкти (посилання копіюються, не об'єкти). Зміна мутабельного значення через клон відобразиться в оригіналі. Для deep copy потрібна ручна реалізація або stream+collect.

[[java-collection-framework-hashmap-replaceall]]
.HashMap.replaceAll() та modCount
`replaceAll(BiFunction)` обходить всю таблицю і для кожного вузла замінює значення результатом функції. Якщо функція повертає те ж саме значення — entries все одно "замінюються" (setValue викликається). `modCount` при `replaceAll` поводиться специфічно: у Java 8-10 він не збільшується під час `replaceAll`, у Java 11+ поведінка може відрізнятися. Безпечно ітерувати через `entrySet()` і використовувати `setValue()` у entry — це не є структурною модифікацією.

.Mutable ключ — втрата елементу

    int id;
    MutableKey(int id) { this.id = id; }

    @Override public int hashCode() { return id; }
    @Override public boolean equals(Object o) { return ((MutableKey)o).id == id; }
}

HashMap<MutableKey, String> map = new HashMap<>();
MutableKey key = new MutableKey(42);

System.out.println(map.get(key)); // "value" ✅

System.out.println(map.size()); // 1 — елемент ТАМ, але ми не можемо його знайти!
----

// НЕПРАВИЛЬНО — не враховує load factor:

.Ітерація під час модифікації — fail-fast
[source,java]
----
Map<String, Integer> map = new HashMap<>();
map.put("a", 1); map.put("b", 2); map.put("c", 3);

// НЕПРАВИЛЬНО — ConcurrentModificationException:
for (Map.Entry<String, Integer> entry : map.entrySet()) {
    if (entry.getValue() == 2) {
        map.remove(entry.getKey()); // ❌ ConcurrentModificationException!
    }
}

// ПРАВИЛЬНО — через Iterator.remove():
Iterator<Map.Entry<String, Integer>> it = map.entrySet().iterator();
while (it.hasNext()) {
    if (it.next().getValue() == 2) {
        it.remove(); // ✅ безпечно
    }
}

// АБО Java 8+ через removeIf:
map.entrySet().removeIf(e -> e.getValue() == 2); // ✅

// АБО collect keys + remove:
map.keySet().removeIf(k -> map.get(k) == 2); // ✅
----

.Одночасна модифікація під час ітерації — ConcurrentModificationException
[source,java]
----
Map<String, Integer> map = new HashMap<>();
map.put("a", 1); map.put("b", 2); map.put("c", 3);

// ПРОБЛЕМА 1 — for-each + map.remove() → ConcurrentModificationException:
for (Map.Entry<String, Integer> entry : map.entrySet()) {
    if (entry.getValue() == 2) {
        map.remove(entry.getKey()); // ❌ modCount змінився → fail-fast!
    }
}

// ПРОБЛЕМА 2 — паралельний потік модифікує map під час ітерації:
// Thread A: iterates map.entrySet()
// Thread B: map.put("d", 4)  ← може спровокувати resize → modCount++ → CME у Thread A

// РІШЕННЯ 1 — Iterator.remove():
Iterator<Map.Entry<String, Integer>> it = map.entrySet().iterator();
while (it.hasNext()) {
    if (it.next().getValue() == 2) {
        it.remove(); // ✅ Iterator сам оновлює modCount
    }
}

// РІШЕННЯ 2 — Java 8 removeIf:
map.entrySet().removeIf(e -> e.getValue() == 2); // ✅

// РІШЕННЯ 3 — зібрати ключі окремо і видалити після ітерації:
List<String> toRemove = map.entrySet().stream()
    .filter(e -> e.getValue() == 2)
    .map(Map.Entry::getKey)
    .collect(Collectors.toList());
toRemove.forEach(map::remove); // ✅
----

WARNING: `ConcurrentModificationException` — це *fail-fast* механізм через лічильник `modCount`. Він не є thread-safe гарантією — це лише best-effort detection. У багатопотоковому коді він може *не* спрацювати. Для concurrent-доступу завжди використовуй `ConcurrentHashMap`.

.Плутанина між initialCapacity та кількістю елементів
[source,java]
----
// ПОШИРЕНА ПОМИЛКА: initialCapacity = очікувана кількість елементів
// При 1000 елементах з new HashMap<>(1000):
// threshold = 1000 * 0.75 = 750 → resize відбудеться вже при 750-му елементі!

// ПРАВИЛЬНА ФОРМУЛА: initialCapacity = expectedElements / loadFactor + 1

map.put(null, 42);
map.put(null, 100); // перезаписує — null ключ унікальний!

System.out.println(map.get(null));        // 100
System.out.println(map.containsKey(null)); // true
System.out.println(map.size());            // 1, не 2

// Пастка: неможливо відрізнити "ключ відсутній" від "ключ є, значення null"
map.put("exists", null);
System.out.println(map.get("exists"));   // null — ключ є зі значенням null
System.out.println(map.get("missing"));  // null — ключ відсутній
// Обидва повертають null! Розрізнити можна тільки через containsKey:
System.out.println(map.containsKey("exists"));  // true
System.out.println(map.containsKey("missing")); // false

// Внутрішньо: null ключ → hash = 0 → завжди у table[0]
----

TIP: *Коли використовувати `HashMap` vs альтернативи:* `HashMap` — коли потрібна швидка (O(1)) несортована Map для одного потоку. `LinkedHashMap` — коли потрібен передбачуваний порядок ітерації або LRU-кеш. `TreeMap` — коли потрібні сортовані ключі або range-запити (`headMap`, `subMap`). `ConcurrentHashMap` — у будь-якому багатопотоковому контексті. `EnumMap` — коли ключі є enum-константами (2-5x швидше). `WeakHashMap` — для кешів де записи мають автоматично видалятися разом з ключами.

.Корисні Java 8+ методи
[source,java]
----
Map<String, Integer> wordCount = new HashMap<>();

// Підрахунок слів — merge (найелегантніший спосіб):
String[] words = {"java", "go", "java", "python", "java"};
for (String word : words) {
    wordCount.merge(word, 1, Integer::sum);
}
// {java=3, go=1, python=1}

// computeIfAbsent — кешування / мультимапа:
Map<String, List<String>> multiMap = new HashMap<>();
multiMap.computeIfAbsent("key", k -> new ArrayList<>()).add("value");

// getOrDefault — уникнення NPE:
int count = wordCount.getOrDefault("scala", 0);

// replaceAll — масове оновлення значень:
wordCount.replaceAll((k, v) -> v * 10);
// {java=30, go=10, python=10}

// putIfAbsent — ініціалізація якщо відсутній:
wordCount.putIfAbsent("rust", 0);
----

.HashMap vs інші реалізації Map
[cols="3,1,1,1,1", options="header"]
|===
|Складність get/put             |O(1)*              |O(1)*              |O(log n)           |O(1)*
|`null` ключ                    |✅ один            |✅ один            |❌ (без Comparator) |❌
|Thread-safe                    |❌                 |❌                 |❌                 |✅
|Пам'ять (vs HashMap)           |baseline           |+16 байт/entry     |+16 байт/entry     |+~20%
|Коли використовувати           |загального призначення |LRU cache, стабільний порядок |відсортований Map |multithreading
|===

IMPORTANT: Вибір реалізації Map: (1) Якщо потрібна швидкість і порядок не важливий → `HashMap`. (2) Якщо потрібен insertion-order або LRU cache → `LinkedHashMap`. (3) Якщо потрібне сортування по ключу → `TreeMap`. (4) Якщо багатопотоковість → `ConcurrentHashMap`. Ніколи не використовуй `Hashtable` у новому коді.

=== Best practices / checklist for reviewers

- Перевірити: використання immutable keys, hashCode/equals contract, initialCapacity, loadFactor.
- Переконатись, що concurrent structures використовуються тільки коли потрібно; прості synchronized блоки можуть бути прийнятні при low concurrency.
- Для кешів — використовувати battle-tested бібліотеки (Caffeine), настроїти weigher та eviction policy; тестувати під production-like load.
- Використовувати JMH для мікробенчмарків критичних hot-paths перед оптимізацією.
