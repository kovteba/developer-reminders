=== Class Thread

.Статичні методи клаусу `Thread`:

* `Thread.currentThread()`;
* <<java-concurrency-sleep,`Thread.sleep(long millis)`>>;
* <<java-concurrency-thread-yield,`Thread.yield()`>>;
* `Thread.interrupted()`;
* `Thread.holdsLock(Object obj)`;
* `Thread.activeCount()`;
* `Thread.dumpStack()`;
* `Thread.onSpinWait()` (Java 9+).

.Нестатичні методи клаусу `Thread`:

Керування та статус:

* `start()`;
* `run()`;
* <<java-concurrency-join,`join()` / `join(long millis)`>>;
* `interrupt()`;
* `isInterrupted()`;
* `isAlive()`;
* `getState()`.

Налаштування та дані:

* `getName()` / `setName(String name)`;
* `getPriority()` / `setPriority(int priority)`;
* `threadId()` (Java 19+) або `getId()` (застарілий);
* `isDaemon()` / `setDaemon(boolean on)`;
* `getStackTrace()`;
* `getThreadGroup()`;
* `setUncaughtExceptionHandler(UncaughtExceptionHandler eh)`;
* `getUncaughtExceptionHandler()`.

Застарілі (Deprecated) — не використовувати:*

* `stop()`;
* `suspend()`;
* `resume()`.

[#java-concurrency-sleep]
.Thread.sleep()
`Thread.sleep()` статичний метод класу `Thread` який зупиняє виконання потоку в якому був викликан. Після виклику цього методу система перестає виділяти процесорний час потоку. Виконується заданий час або зупинений прериванням. Після виклику метода поток має стан <<java-concurrency-thread-life-circle,TIMED_WAITING>>. Після закінчення цього методу поток повертається до пулу зі статусом <<java-concurrency-thread-life-circle,RUNNABLE>>.

[#java-concurrency-thread-yield]
.Thread.yield()
Статичний метод `Thread.yield()` змушує процесор перемикнутися на обробку іншого потоку. Це може бути зручно якщо поток чекає на якусь умову і необхідно що частіше перевіряти її викликаючі цей метод у циклі. Цей метод намагається сказати планувальнику що треба виконати інший поток що є в черзі на виконання. Виклик цього методу намагається перевести поток з *Running* у *Runnable*. Але виклак цього методу може і не призвести до бажаного результату.

[#java-concurrency-join]
.Thread.join()
В Java є засоби які дозволяють чекати один поток закінчення іншого. Як що нам треба щоб головний поток чекав завершення побочного потоку, для цього необхідно із головного потому виклакати метод `join()` на єкземплярі побочного потоку. У цього випадку як тільки побочний поток завершиться метод `join()` поверне керування головному потоку і він зможе пробовжити виконання. `join()` має перевантажену версію, в якій ми можемо вказати час очікування. у даному випадку керування повернеться до головного потому або коли побочний закінчиться, або коли закінчиться час очікування. Поток у якому викликається цей метод переходить у статус `BLOCKED`.

[#java-concurrency-thread-life-circle]
.Thread life circle
Поток можу бути в таких станах: `NEW`, `RUNNABLE`, `RUNNING`, `BLOCKED`, `TERMINATED`, `WAITING`, `TIMED_WAITING`.

image::{concurrencyimgsdir}/threadlife.png[scaledwidth=75%]

1. *New*. Коли створюємо єкземпляр класу Thread за допомогою слова `new`. в цьому стані поток ще не працює;
2. *Runnable*. Коли викликаємо метод `start()` поток буде мати такий стан. Але потік ще не працює, тому що він спочатку потік передається планіровщіку потоків (Thread scheduler). Чи запусте планіровщік поток одразу чи ні залежить від реалізаціє його в ОС. Поток може перейти у цей стан зі станів *Rinnung* та *Blocked*;
3. *Running*. Поток буде мати такий стан коли він буде запущений планувальником потоків. Планувальник обирає один з потоків і переводе у цей стан, і процесор одразу починає виконувати поток. Стан потоку може змінитися на *Runnable*, *Dead* или *Blocked*;
4. *Blocked* або *Waiting*. Поток може чекати на інщий поток поки він закінче свою роботу або ждати якісь загальні ресурси. У цьому випадку стан потоку буде *Waiting*. Після скінчення очікування його стан зміниться на *Runnable* і він повернеться в загальний пул потоків;
5. *Dead*. Після завершя роботи поток буде мати такий стан. Поток з таким стано не може бути заново запущен.

[#java-concurrency-deamon-thread]
.Daemon thread
__Потоками-демонами__ - є потоки які працюють у фоновому режимі. У Java процес завершується коли завершується останій поток. Навіть якщо `main()` вже закінчився але працює хочаб одни поток створений їм система буде чекати завершення. Однак це не відноститься до потоків демонів. Якщо останій звичайний поток буде завершений а лишаться лише демони то вону будуть примусово припинені. Для створення потока демоне необхідно перед запуском потоку викликати метод `setDaemon(true)`. Нижче наведен приклад. Ми створимо клас `CounterThread` який буде просто робити якусь роботу і зараз це для нас не важливо.

Як бачимо демон ніколи не буде завершатися томощо головні треди вже закінчилися і система не чікає завершення демонів. Тому використовують їх як допоміжні потоки для звичайних.

Якщо потік демона в Java створює інший потік, цей дочірній потік успадкує статус демона свого батьківського потоку. Демо поток можна використовувати для обробки безпрерервних фонових завдань тому щопоток демон не перешкоджатиме завершенню програми після завершення основного потоку. Яле в той самий час використання демон потоків для завдань які пов'язані за записом у файл може бути ризикованим, тому що коли закінчеться виконання головного потоку демон буде раптово припинено і тому файл може бути пошкодженим.

[#java-concurrency-thread-dump]
.Дамп thread
Дамп дозволяє дізнатися які потокі працюють. Для того щоб отримати цю інформацію можна використати у linux `kill -3`, або утилітою `jstack` а для отримання PID процесу `jps`.
