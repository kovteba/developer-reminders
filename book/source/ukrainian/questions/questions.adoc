// - Що таке Spliterator і чим він кращий за Iterator? Відповідь: Він дозволяє розділяти джерело даних на частини для паралельної обробки (trySplit()) і повідомляє про характеристики джерела (characteristics()).
// - Які бувають характеристики (characteristics) у Spliterator? Відповідь: ORDERED, DISTINCT, SORTED, SIZED, NONNULL, IMMUTABLE, CONCURRENT, SUBSIZED.
// - Як Stream дізнається, що можна оптимізувати count()? Відповідь: Якщо Spliterator має характеристику SIZED, count() поверне розмір джерела без ітерації.
// - Що відбувається при виклику sorted() у середині стріма? Відповідь: Це "Stateful" операція. Sink накопичує всі елементи в список, сортує їх у методі end(), і тільки потім пускає далі.
// - Чи можна перевикористати стрім? Чому виникає IllegalStateException? Відповідь: Ні. Стрім — це одноразовий ітератор по пайплайну. Після виклику термінальної операції він позначається як "consumed".
// - Як працює Stream.concat під капотом? Відповідь: Створюється новий Spliterator, який спочатку вичитує перший стрім, а потім другий. Закриття результуючого стріма закриває обидва вхідні.
// - Яка роль пакета java.util.stream у модулі java.base? Відповідь: Це ядро стрімів, де лежать базові абстракції (AbstractPipeline, ReferencePipeline).

// .Блок 2: Паралельні стріми (Parallelism & Concurrency)
// - Який Thread Pool використовують Parallel Streams? Відповідь: Загальний ForkJoinPool.commonPool().
// - В чому небезпека використання parallelStream() у веб-серверах (напр. Spring Boot)? Відповідь: Можна забити commonPool, що сповільнить роботу інших частин системи (напр. обробку запитів).
// - Як змінити кількість потоків для конкретного паралельного стріма? Відповідь: Виконати його всередині кастомного ForkJoinPool.submit(...).
// - Коли parallel() зробить код повільнішим? Відповідь: На малих даних, на нелінійних структурах (LinkedList), або коли операції занадто легкі (overhead на поділ/злиття більший за корисну роботу).
// - Як parallelStream працює з ORDERED характеристикою? Відповідь: Він намагається зберегти порядок, але це вимагає синхронізації (buffer), що сильно знижує продуктивність.
// - Різниця між reduce та collect у паралельному стрімі? Відповідь: reduce створює нові об'єкти (immutable), collect модифікує існуючий контейнер (mutable), що ефективніше для колекцій.
// - Чому findAny() краще за findFirst() у паралельних стрімах? Відповідь: findAny повертає перший ліпший результат з будь-якого потоку, а findFirst змушений чекати результатів з "початку" джерела.
// - Як Spliterator ділить LinkedList для паралельного стріма? Відповідь: Дуже погано. Йому доводиться ітерувати до середини, щоб знайти точку розриву (O(n)).
// - Чи безпечно змінювати колекцію-джерело під час виконання стріма? Відповідь: Ні, це призведе до ConcurrentModificationException (non-interference rule).
// - Що таке Combiner у методі reduce? Відповідь: Функція, яка об'єднує результати обробки двох різних гілок (threads) у паралельному стрімі.
// .Блок 3: Collectors & Reduction (Глибина)
// - Як працює Collectors.groupingBy? Відповідь: Створює Map, де ключі — результат функції, а значення — список елементів.
// - Різниця між groupingBy та partitioningBy? Відповідь: Partitioning — це спеціальний випадок grouping, де ключ завжди Boolean. Він швидший.
// - Як зробити groupingBy у ConcurrentMap? Відповідь: Використати groupingByConcurrent.
// - Що робить Collectors.collectingAndThen? Відповідь: Дозволяє виконати фінальну дію над результатом колектора (напр. зробити список Unmodifiable).
// - Як працює teeing колектор (Java 12+)? Відповідь: Він направляє стрім у два різні колектори і потім об'єднує їхні результати.
// - Чи можна написати свій Collector? Які методи треба реалізувати? Відповідь: Так. supplier(), accumulator(), combiner(), finisher(), characteristics().
// - Навіщо потрібен identity у reduce? Відповідь: Це початкове значення і результат для порожнього стріма.
// - Чому Optional<T> reduce(BinaryOperator<T> accumulator) не має identity? Відповідь: Бо без початкового значення він не знає, що повернути для порожнього стріма, тому повертає Optional.empty().
// - Що таке Identity finish у Collector? Відповідь: Характеристика, яка каже, що результат accumulator не потребує трансформації у finisher.
// - Яка проблема з Collectors.toList() у порівнянні з Stream.toList() (Java 16+)? Відповідь: Collectors.toList() зазвичай повертає мутабельний ArrayList, а Stream.toList() — незмінний список.
// .Блок 4: Продуктивність та Пам'ять
// - Як уникнути боксингу (Boxing) у стрімах? Відповідь: Використовувати примітивні стріми: IntStream, LongStream, DoubleStream.
// - Що ефективніше: filter().findFirst() чи цикл for з if та break? Відповідь: Цикл зазвичай трохи швидший через відсутність об'єктів стріма, але на великих даних різниця нівелюється JIT-компілятором.
// - Чи закриває стрім ресурси автоматично? Відповідь: Ні (крім flatMap для внутрішніх стрімів). Треба використовувати try-with-resources.
// - Як Stream.distinct() працює з великими обсягами даних? Відповідь: Він створює внутрішній HashSet. Якщо даних багато — це може призвести до OutOfMemoryError.
// - Чи впливає порядок filter() та map() на продуктивність? Відповідь: Так. Завжди краще спочатку фільтрувати, щоб зменшити кількість об'єктів для мапінгу.
// - Що таке "Lazy Evaluation" і як вона допомагає економити CPU? Відповідь: Обробка починається тільки при виклику термінальної операції. Дані обробляються "пайпами", а не копіями.
// - Як працює limit() у нескінченних стрімах? Відповідь: Це short-circuiting операція. Як тільки лічильник досягає межі, вона посилає сигнал зупинки.
// - Чи є різниця між list.stream().forEach() та list.forEach()? Відповідь: Так. Другий варіант зазвичай швидший, бо працює напряму через ітератор колекції без створення Pipeline об'єктів.
// - Як peek() впливає на продуктивність? Відповідь: Він додає ще один виклик методу на кожен елемент. Це чисто для дебагу.
// - Що буде, якщо в стрімі використати parallel() і sequential() кілька разів? Відповідь: Виграє останній виклик. Весь стрім буде або паралельним, або послідовним.
// .Блок 5: Java 9, 10, 11... 21 Updates (Еволюція Stream API)
// - Чим Stream.ofNullable() (Java 9) корисний при роботі з Map? Відповідь: Він дозволяє уникнути NullPointerException або ручної перевірки, перетворюючи null у порожній стрім.
// - Різниця між limit(n) та takeWhile(predicate) (Java 9)? Відповідь: limit відсікає за кількістю, а takeWhile зупиняє стрім, як тільки елемент перестав відповідати умові (важливо для відсортованих даних).
// - Як працює dropWhile? Відповідь: Він ігнорує елементи, поки предикат true, і починає пропускати все підряд, як тільки зустрів перший false.
// - Навіщо додали Stream.iterate з предикатом (Java 9)? Відповідь: Це аналог циклу for(int i=0; i<n; i++), що дозволяє створювати скінченні стріми без limit.
// - Що таке Collectors.filtering та flatMapping (Java 9)? Відповідь: Це "downstream" колектори. Вони дозволяють фільтрувати або розгортати дані всередині groupingBy, щоб не втратити порожні групи (ключі).
// - Яка особливість Stream.toList() (Java 16)? Відповідь: Це не просто цукор для .collect(Collectors.toList()). Він повертає незмінний список (unmodifiable) і працює швидше, бо йому не потрібен Collector як посередник.
// - Як працює Optional.stream() (Java 9)? Відповідь: Перетворює Optional у стрім з 0 або 1 елемента. Це ідеально для flatMap(Optional::stream).
// - Що таке Collectors.teeing (Java 12) і де його застосувати? Відповідь: Дозволяє обчислити два різні показники за один прохід (наприклад, суму та кількість одночасно), щоб потім порахувати середнє.
//  - Чи є зміни в Stream API у зв'язку з Virtual Threads (Java 21)? Відповідь: Прямих змін в API немає, але parallelStream() тепер може бути менш вигідним, ніж запуск звичайних стрімів у тисячах Virtual Threads через ExecutorService.
// - Як Stream.gatherers() (Java 22/23 Preview) розширює можливості? Відповідь: Це найгучніше оновлення. Воно дозволяє створювати кастомні проміжні оператори (як scan, windowFixed тощо), чого раніше не можна було зробити без втручання в нутрощі JDK.
// .Блок 6: Обробка винятків (Exceptions) та Side-effects
// Чи можна викинути Checked Exception всередині .map()? Відповідь: Ні, функціональні інтерфейси (Function, Predicate) не оголошують throws. Потрібно обертати в try-catch або використовувати wrapper.
// Як працює "Lombok @SneakyThrows" зі стрімами? Відповідь: Він обманює компілятор, дозволяючи викидати checked exceptions як unchecked, але це вважається поганою практикою.
// Що станеться з Parallel Stream, якщо один потік викине RuntimeException? Відповідь: Стрім негайно припинить обробку, і виняток прокинеться в основний потік (той, що викликав термінальну операцію).
// Чому Side-effects (зміна зовнішніх змінних) у стрімах — це зло? Відповідь: Це ламає детермінізм. У паралельних стрімах це призведе до Race Condition, якщо тільки ти не використовуєш Atomic змінні або Concurrent колекції.
// Чи можна використовувати Stream для запису в базу даних через forEach? Відповідь: Можна, але це антипатерн. Краще зібрати дані в пачки (batches) через collect і зберегти одним запитом.
// .Блок 7: Optional та Stream API (Тандем)
// - Різниця між Optional.ifPresent() та Stream.forEach()? Відповідь: Перший працює з 0-1 значенням, другий — з будь-якою кількістю. ifPresent — це термінальна дія для Optional.
// - Як зробити flatMap по Optional полю в об'єкті? Відповідь: .flatMap(obj -> obj.getOptionalField().stream()).
// - Чому Optional не є серіалізованим (Serializable)? Відповідь: Він створювався як повертаємий тип для методів, а не для зберігання в полях класів (DTO/Entity).
// - Коли варто використовувати orElseGet() замість orElse()? Відповідь: Завжди, коли обчислення дефолтного значення є дорогим (наприклад, запит в БД), бо orElse виконується завжди, навіть якщо значення є.
// - Що поверне stream.findAny().orElse(null)? Відповідь: Значення або null, але це "запах коду". Краще продовжувати ланцюжок через Optional.
// .Блок 8: Stateful vs Stateless та Складні оператори
// - Які оператори є Stateful? Відповідь: sorted(), distinct(), limit(), skip(). Вони потребують знання про інші елементи стріма.
// - Чому distinct() на нескінченному стрімі — це проблема? Відповідь: Бо внутрішній HashSet буде рости нескінченно, поки не впаде OOM.
// - Як limit() працює разом із sorted()? Відповідь: Це "найгірший" випадок. Стрім змушений відсортувати все, і тільки потім віддати перші N.
// - Чи гарантує forEachOrdered() порядок у паралельному стрімі? Відповідь: Так, але це вбиває весь профіт від паралелізму, бо елементи будуть чекати своєї черги на вихід.
// - Як працює reduce з трьома аргументами? Відповідь: identity (початкове), accumulator (додавання в поточному потоці), combiner (злиття результатів потоків).
// .Блок 9: Тонкощі Spliterator та Колекторів
// - Як Spliterator оцінює розмір через estimateSize()? Відповідь: Це приблизне значення. Для ArrayList воно точне, для Files.lines() — ні.
// - Що робить характеристика CONCURRENT у Spliterator? Відповідь: Вказує, що джерело може безпечно модифікуватися під час ітерації іншими потоками.
// - Різниця між Characteristics.IDENTITY_FINISH та іншими у Collector? Відповідь: Якщо вона є, результат акумулятора каститься до результату колектора напряму, без виклику finisher().
// - Як працює Collectors.joining() всередині? Відповідь: Використовує StringBuilder для конкатенації рядків.
// - Чому Collectors.toMap кидає Exception при однакових ключах? Відповідь: За замовчуванням він не знає, яке значення вибрати. Треба передати третій аргумент — mergeFunction.
// .Блок 10: JIT, Оптимізації та "Під капотом"
// - Як JIT-компілятор (C2) ставиться до довгих ланцюжків Stream API? Відповідь: JIT добре справляється з "інлайнінгом" (inlining) лямбд, але довгі ланцюжки створюють багато об'єктів Sink. Якщо стрім невеликий, іноді він не встигає оптимізуватися, і звичайний for виграє.
// - Що таке "Stream Fusion"? Відповідь: Це термін, що описує здатність бібліотеки об'єднувати кілька операцій (наприклад, два filter поспіль) в один прохід по даних. В Java це реалізовано через ланцюжок Sink.accept().
// - Як IntStream допомагає уникнути витоків пам'яті в порівнянні з Stream<Integer>? Відповідь: Окрім відсутності боксингу, примітивні стріми споживають значно менше пам'яті (у 3-4 рази), що зменшує навантаження на Garbage Collector.
// - Що станеться, якщо змінити джерело стріма (наприклад, ArrayList) під час виконання в іншому потоці? Відповідь: Більшість Spliterator-ів кинуть ConcurrentModificationException через механізм fail-fast, перевіряючи modCount.
// - Як працює Stream.empty()? Чи створює він нові об'єкти? Відповідь: Ні, він використовує статичний синглтон, що робить його дуже дешевим.
// .Блок 11: Порівняння та архітектурний вибір
// - Java Streams vs Kotlin Sequences: головна різниця? Відповідь: Sequences в Kotlin завжди послідовні (немає вбудованого parallel()), але вони мають менший overhead і більш лаконічний синтаксис для обробки винятків.
// - В яких випадках RxJava або Project Reactor кращі за Stream API? Відповідь: Коли потрібна асинхронність, обробка подій у часі, backpressure або робота з нескінченними потоками даних у реактивному стилі. Stream API — для обробки даних у пам'яті "тут і зараз".
// - Чи варто зберігати Stream як поле класу? Відповідь: Категорично ні. Стрім — це стан (stateful object) процесу обчислення, він не серіалізується і "протухає" після використання.
// - Чому в Stream API немає методу zip (об'єднання двох стрімів попарно)? Відповідь: Це складно реалізувати для паралельних стрімів, де частини даних знаходяться в різних потоках. (З'явиться в Gatherers у нових Java).
// - Як реалізувати пагінацію за допомогою skip() та limit()? Які ризики? Відповідь: Це працює, але для великих значень skip(1000000) стрім все одно має проітерувати (відкинути) мільйон елементів, що неефективно для БД.
// .Блок 12: Складні Case-studies
// - Як за допомогою стрімів реалізувати "sliding window" (ковзне вікно)? Відповідь: До Java 22 це робилося через IntStream.range та subList, або через сторонні бібліотеки (Vavr, StreamEx). Тепер — через Gatherers.windowFixed/Sliding.
// - Як знайти дублікати в стрімі за допомогою filter? Відповідь: set = new HashSet(); ... .filter(n -> !set.add(n)). Але це side-effect, який зламає паралельний стрім!
// - Як реалізувати власну термінальну операцію? Відповідь: Тільки через .collect(Collector) або .reduce(). Справжні кастомні термінальні операції потребують реалізації внутрішніх інтерфейсів JDK.
// - Що робити, якщо flatMap повертає занадто багато даних і виникає OOM? Відповідь: Переходити на реактивні стріми з підтримкою backpressure або обробляти дані частинами (batches).
// - Як перевірити, чи стрім паралельний, всередині лямбди? Відповідь: Викликати Thread.currentThread().getName() — якщо там ForkJoinPool.commonPool-worker-..., то це паралельна гілка.
// .Блок 13: Питання на "засипку" (Corner cases)
// - Чи може stream.map(x -> x).count() бути оптимізований до size()? Відповідь: Ні. Оскільки map може змінити кількість елементів (якщо там кидається Exception) або тип, Java змушена виконати пайплайн.
// - Що поверне Stream.iterate(0, i -> i + 1).filter(i -> i < 0).findFirst()? Відповідь: Програма зависне в нескінченному циклі, намагаючись знайти від'ємне число в ітераторі, що зростає.
// - Яка різниця між Stream.builder() та Stream.of()? Відповідь: Builder корисний, коли ти додаєш елементи в циклі (логіка накопичення), а of — коли дані вже готові.
// - Чи можна зробити flatMap по Map? Відповідь: Тільки через map.entrySet().stream().flatMap(...).
// - Як перетворити Stream<String> у String[]? Відповідь: .toArray(String[]::new). Важливо використовувати посилання на конструктор масиву.
// .Блок 14: Філософія та Best Practices
// - Коли Java Stream — це "антипатерн"? Відповідь: Коли логіка всередині лямбд занадто складна (багато рядків, вкладені try-catch), або коли звичайний for набагато читабельніший.
// - Що таке "Fluent API" в контексті стрімів? Відповідь: Це стиль проектування, де кожен метод повертає об'єкт того ж (або спорідненого) типу, дозволяючи будувати ланцюжки.
// - Як стріми допомагають у написанні "чистого коду"? Відповідь: Вони переводять акцент з "Як робити" (цикли, індекси) на "Що робити" (фільтрувати, перетворювати).
// - Чому не рекомендується використовувати parallelStream для IO-задач? Відповідь: Бо commonPool має обмежену кількість потоків (зазвичай рівну кількості ядер CPU). IO-задачі блокують ці потоки, не даючи CPU виконувати корисну роботу.
// - Чи є в стрімах механізм "Retry" (повтор при помилці)? Відповідь: Ні, стандартний Stream API не підтримує повторних спроб.
// .Блок 15: Фінальні штрихи (Java 21+)
// - Що таке SequencedCollection і як вона впливає на стріми? Відповідь: Вона гарантує порядок зустрічі елементів, що робить операції findFirst та reverse більш передбачуваними.
// - Як працювати з Scoped Values у стрімах? Відповідь: Це складно, оскільки лямбди можуть виконуватися в різних потоках (parallel), де Scoped Value може бути недоступним.
// - Чи можна використовувати Stream для генерації послідовностей Фібоначчі? Відповідь: Так, через Stream.iterate(new int[]{0, 1}, t -> new int[]{t[1], t[0] + t[1]}).
// - Яка роль java.util.stream.Collector.Characteristics.CONCURRENT? Відповідь: Вона дозволяє одному об'єкту-результату (напр. ConcurrentHashMap) приймати дані з різних потоків паралельно без виклику combiner.
// - Найголовніше питання: Плов чи Java? Відповідь: Правильний Senior спочатку ставить казан на вогонь (на ту саму підставку 32 см), а поки плов мліє — пише ідеальний код на стрімах.
